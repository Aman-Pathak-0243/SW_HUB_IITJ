# Decision Log

Every **major technical decision** is recorded here so future student developers
understand not just *what* the system does but *why*. Append-only: add new
records; correct an old one with a follow-up record rather than rewriting it.
Each entry follows: **Decision · Alternatives considered · Why chosen ·
Trade-offs · Future impact**.

> Updated at the end of **every session** (per [SESSION_PROTOCOL.md](SESSION_PROTOCOL.md)).
> The lighter, narrative ADRs in [ARCHITECTURAL_DECISIONS.md](ARCHITECTURAL_DECISIONS.md)

## Session 1 — Architecture & schema decisions

## DL-001 — Deliver V2 across 10 structured Claude Code sessions, with documentation as a living, always-current artifact.

- **Decision:** Deliver V2 across 10 structured Claude Code sessions, with documentation as a living, always-current artifact.
- **Alternatives considered:** One long build; or ad-hoc sessions without a fixed handoff protocol.
- **Why this was chosen:** A rotating, single-institute team needs a deterministic handoff: each session reads the tracking files, continues from the next pending task, and updates docs before ending. See SESSION_PROTOCOL.md.
- **Trade-offs:** Upfront protocol overhead and discipline to keep docs in lockstep with code.
- **Future impact:** Any future student can resume mid-project with zero tribal knowledge; sessions never repeat completed work.

## DL-002 — Store all credentials (Neon DATABASE_URL, Mongo URI, OAuth secrets) only in git-ignored .env.local; redact them in all tracked docs.

- **Decision:** Store all credentials (Neon DATABASE_URL, Mongo URI, OAuth secrets) only in git-ignored .env.local; redact them in all tracked docs.
- **Alternatives considered:** Commit a configured .env; paste connection strings into docs/runbooks.
- **Why this was chosen:** Secrets must never enter version control (the V1 README leak is the cautionary tale). .env* is git-ignored; env.example holds placeholders only.
- **Trade-offs:** Each environment must be configured out-of-band; onboarding needs the secret-sharing step documented.
- **Future impact:** Eliminates a whole class of leaks; the gitleaks CI (Session 1) enforces it.

## DL-003 — Produce a verified, self-contained backup (source + /public + full Mongo dump) BEFORE any migration, and re-verify by re-extraction.

- **Decision:** Produce a verified, self-contained backup (source + /public + full Mongo dump) BEFORE any migration, and re-verify by re-extraction.
- **Alternatives considered:** Trust git + Atlas as the backup; skip a pre-migration snapshot.
- **Why this was chosen:** The master spec forbids modifying data before a verified backup. The Mongo dump also captured a previously-undocumented `queries` collection that git could not have preserved.
- **Trade-offs:** A ~73MB local artifact per run (git-ignored); the live DB dump needs credentials.
- **Future impact:** Migrations (Sessions 5–7) can proceed safely with a known-good restore point.

## DL-004 — Composition rule for the three orthogonal axes (academic-year scope x draft/publish x version history), and the explicit scope of versioning.

- **Decision:** Composition rule for the three orthogonal axes (academic-year scope x draft/publish x version history), and the explicit scope of versioning.
- **Alternatives considered:** (a) Generic content_entity + content_field_value EAV + two version pointers. (b) Polymorphic content_version holding a full JSONB snapshot keyed by (entity_type, entity_id). (c) content_item header + immutable content_revision body + per-type *_payload tables 1:1 with revision.
- **Why this was chosen:** Adopted (c). Composition rule: YEAR scopes the header (academic_year_id on content_item/org_unit/appointment); LIFECYCLE (content_status) lives on the header plus two cache pointers; VERSIONS are immutable children of the header. Versioning (capability 7) is EXPLICITLY scoped to CMS payload content only. org_unit and appointment participate in the draft/publish gate but are NOT versioned: their cross-year history is per-year rows tied by org_unit_lineage, and their intra-year edit history is audit_log only (no restore). This removes the prior asymmetry/contradiction where structural rows had a status but no revision lineage.
- **Trade-offs:** Structural/roster edits within a year are not point-in-time restorable (only audited). Per-type payload tables mean a 1:1 index-only join per content read.
- **Future impact:** New module = one content_type_def row + one payload table; the spine grants year-scope, draft/publish, versioning, media, and audit for free. Maintainers have one clear rule for what is versioned vs audit-only.

## DL-005 — Remove the circular content_item<->content_revision deferrable FK; make published_revision_id / draft_revision_id PLAIN non-FK cache columns guarded by partial uniques + a trigger.

- **Decision:** Remove the circular content_item<->content_revision deferrable FK; make published_revision_id / draft_revision_id PLAIN non-FK cache columns guarded by partial uniques + a trigger.
- **Alternatives considered:** (a) Keep the circular FK as DEFERRABLE INITIALLY DEFERRED (Prisma-hostile raw migration). (b) Derive published/draft purely via indexed queries with no cache columns. (c) Plain cache columns, no FK, integrity via constraints+trigger.
- **Why this was chosen:** Adopted (c). The circular deferrable FK was the single most student-hostile, Prisma-invisible piece of the schema. Plain uuid cache columns avoid the deferrable migration entirely; correctness is backed by content_revision partial uniques (at most one draft AND one published per item) and a trigger asserting each pointer references a same-item revision of the matching status. The pointers are always rebuildable from content_revision (WHERE content_item_id=? AND revision_status=?).
- **Trade-offs:** Two cache columns can in principle be stale if the trigger is dropped; mitigated by a rebuild script and the documented derive-from-revision fallback. The cache is not a real FK so Prisma models them as scalar uuids.
- **Future impact:** A rotating undergraduate never meets a deferred constraint. The content-spine maintainer doc explains item vs revision vs payload and the pointer semantics in ~10 lines + a diagram.

## DL-006 — content_type as a text-PK LOOKUP TABLE (content_type_def), NOT a Postgres enum.

- **Decision:** content_type as a text-PK LOOKUP TABLE (content_type_def), NOT a Postgres enum.
- **Alternatives considered:** (a) Postgres enum content_type (type-safe, but ALTER TYPE ADD VALUE cannot be added AND used in one transaction/migration). (b) Enum + mandatory two-step migration. (c) Lookup table keyed by text.
- **Why this was chosen:** Adopted (c). content_type is relationally referenced by content_item, so by the schema's own enum-vs-table rule (relationally-referenced/extensible sets => tables) it must be a table. This makes capability-9 module addition pure DATA + a new payload table in a single ordinary migration, eliminating the enum-in-transaction failure that would break the headline extensibility story.
- **Trade-offs:** Loses compile-time enum exhaustiveness in Prisma; mitigated by an in-code config map (content_type -> payload table + handler) plus a startup test asserting every content_type_def row is handled. Payload routing keys off the text discriminator.
- **Future impact:** Adding 'gallery' or 'notice_board' is an admin/dev data insert + one table, never a type migration. Removes the most fragile feasibility blocker.

## DL-007 — Cross-year org-unit identity via a first-class org_unit_lineage table; org_unit.lineage_key and role_assignment.org_unit_lineage_key are REAL FKs.

- **Decision:** Cross-year org-unit identity via a first-class org_unit_lineage table; org_unit.lineage_key and role_assignment.org_unit_lineage_key are REAL FKs.
- **Alternatives considered:** (a) 'Logical FK' to org_unit.lineage_key (non-unique per-year column, no real FK, app-enforced). (b) org_unit as timeless identity + org_unit_year_profile per year. (c) Dedicated org_unit_lineage identity table.
- **Why this was chosen:** Adopted (c). The prior logical FK referenced a non-unique column and could never be a real constraint, leaving security-critical RBAC scope (role_assignment) able to dangle. A tiny org_unit_lineage table (lineage_key PK) owns the identity; every org_unit row and every unit-scoped grant FK into it (ON DELETE RESTRICT). The Transition Wizard reuses an existing lineage row rather than copying a bare uuid.
- **Trade-offs:** One lightweight insert per genuinely new logical unit in the wizard; one extra small table. Negligible at this scale (≈40 units).
- **Future impact:** Converts a class of silent app-layer bugs (typo'd / orphaned lineage_key, wizard mis-copy) into database-enforced integrity — a major maintainability win for rotating maintainers. Real Prisma relations for cross-year queries.

## DL-008 — DB-guard the appointment.org_unit_id denormalization (composite FK + type trigger) and change it to ON DELETE RESTRICT.

- **Decision:** DB-guard the appointment.org_unit_id denormalization (composite FK + type trigger) and change it to ON DELETE RESTRICT.
- **Alternatives considered:** (a) Plain FK to org_unit.id with app-only year/type agreement and ON DELETE CASCADE. (b) Fully normalized (derive unit via position). (c) Composite FK enforcing year agreement + type-compatibility trigger + RESTRICT.
- **Why this was chosen:** Adopted (c). org_unit gains UNIQUE(id, academic_year_id) so appointment FK(org_unit_id, academic_year_id) guarantees the cached unit and the appointment's year cannot disagree. appointment.org_unit_type_id + a trigger ties position.applies_to_type_id so a Warden cannot be appointed to a club. CASCADE was changed to RESTRICT to honor the no-orphan-history policy that every other historized FK follows.
- **Trade-offs:** Slightly wider appointment row (echoed academic_year_id/org_unit_type_id) and a trigger for type-compat. Acceptable for the integrity gained on the hottest roster path.
- **Future impact:** The denormalized hot-read column can no longer drift; roster history is structurally un-erasable.

## DL-009 — Enforce roster cardinality (position.max_holders) at the DB level.

- **Decision:** Enforce roster cardinality (position.max_holders) at the DB level.
- **Alternatives considered:** (a) max_holders advisory / app-only. (b) Partial unique for the singleton case only. (c) Partial unique for singletons + deferred constraint trigger counting active appointments for multi-holder positions.
- **Why this was chosen:** Adopted (c). Singleton positions (secretary, PIC, warden) get a partial UNIQUE(academic_year_id, org_unit_id, position_id) WHERE active. Multi-holder positions (coordinators) are bounded by a deferred constraint trigger comparing the active-appointment count against position.max_holders. max_holders is therefore enforced, not decorative.
- **Trade-offs:** A trigger is more moving parts than a constraint; documented as trigger-enforced in the maintainer doc.
- **Future impact:** Capability-4 cardinality rules ('single secretary, multiple coordinators') are guaranteed even against direct DB writes.

## DL-010 — Give announcement its own announcement_payload table (do not share event_payload).

- **Decision:** Give announcement its own announcement_payload table (do not share event_payload).
- **Alternatives considered:** (a) Two discriminators (event, announcement) sharing event_payload. (b) Collapse to one 'post' content_type with a sub-kind. (c) Separate announcement_payload table.
- **Why this was chosen:** Adopted (c) to honor the schema's own one-shape-one-table principle and avoid the latent divergence trap of two content_types sharing one payload table (the worst of both worlds). Each content_type now maps 1:1 to a payload table, keeping the capability-9 routing function clean.
- **Trade-offs:** A near-duplicate small table now; if events and announcements truly never diverge this is mild redundancy.
- **Future impact:** When announcements gain unique fields (e.g. severity, acknowledgement) no painful table split or meaningless nullable columns are needed.

## DL-011 — Drop content_type_def.field_schema / the generic field-driven editor abstraction.

- **Decision:** Drop content_type_def.field_schema / the generic field-driven editor abstraction.
- **Alternatives considered:** (a) Keep field_schema JSONB (a Zod-like mini-framework) + a generic form renderer. (b) Drop it; use a small in-code config map and typed per-type editors.
- **Why this was chosen:** Adopted (b). The generic field_schema editor was speculative generality for a ~12-type single-institute portal and a third source of truth (enum/def/code) prone to drift. content_type_def keeps only the small behavioral flags (is_year_scoped, supports_publish, is_org_bound) + payload_table; payload shape lives in the typed payload tables and an in-code routing/handler map.
- **Trade-offs:** No runtime-defined arbitrary content forms; adding a module needs a dev (a new payload table + handler), which is the correct safety boundary anyway.
- **Future impact:** Two sources of truth (def row + code handler), reconciled by a startup test asserting every content_type_def row has a handler. Much lower cognitive load for maintainers.

## DL-012 — Centralize audit_log writing in one Prisma client extension / service wrapper.

- **Decision:** Centralize audit_log writing in one Prisma client extension / service wrapper.
- **Alternatives considered:** (a) Per-route/per-service audit calls scattered across the codebase. (b) A single Prisma client extension (or one audited-mutation service) that records before/after for every create/update/delete/publish/transition/grant.
- **Why this was chosen:** Adopted (b). Scattered audit calls inevitably get half-implemented (some tables audited, some not), which is worse than a clear policy. One choke point makes capability-8 coverage uniform and automatic for future modules.
- **Trade-offs:** The extension must capture before-images (an extra read on update/delete) and is a single point that must be maintained.
- **Future impact:** New modules are audited 'for free'; the audited-action list is documented once in the maintainer doc.

## DL-013 — NextAuth v4 PrismaAdapter contract: add VerificationToken, adopt canonical adapter model shape, choose JWT session strategy (drop the DB session table).

- **Decision:** NextAuth v4 PrismaAdapter contract: add VerificationToken, adopt canonical adapter model shape, choose JWT session strategy (drop the DB session table).
- **Alternatives considered:** (a) Custom adapter over bespoke app_user/auth_account/auth_session. (b) Canonical @next-auth/prisma-adapter models (User/Account/VerificationToken) with @map to snake_case + JWT sessions. (c) Database session strategy.
- **Why this was chosen:** Adopted (b). next-auth@4.24.14 requires a VerificationToken model (createVerificationToken/useVerificationToken) and adapter-canonical Prisma field names (userId, providerAccountId). app_user maps to the Adapter User (email, emailVerified via @map, image via avatar_media_id). Session strategy is JWT, so the auth_session table is DROPPED as dead weight; auth_account uses adapter-canonical fields incl. session_state/refresh_token.
- **Trade-offs:** JWT sessions cannot be server-revoked instantly (mitigated by short token TTL + the role_assignment revoked_at check on sensitive actions). Adapter field names diverge from pure snake_case, bridged by @map.
- **Future impact:** The first migration produces a valid, adapter-compatible auth surface; Google OAuth + credentials + email verification all work out of the box.

## DL-014 — Enum vs lookup table policy (with content_type re-classified).

- **Decision:** Enum vs lookup table policy (with content_type re-classified).
- **Alternatives considered:** All enums; all lookup tables; or a mixed rule.
- **Why this was chosen:** Rule: admin/dev-extensible OR relationally-FK-referenced sets => LOOKUP TABLES (org_unit_type, role, permission, position, content_type_def). Small, fixed, code-coupled, non-FK-referenced sets => Postgres ENUMs (all *_status, person_type, meal_type, audience_type, resource_kind, media_kind, media_role, audit_action, storage_provider, transition_status). content_type moved to the table side because it is FK-referenced by content_item.
- **Trade-offs:** Lookup tables need seed rows and lose compile-time exhaustiveness; enums need a migration to extend.
- **Future impact:** Capabilities 2/3/4/9 (orgs, roles, positions, modules) are pure data with no migrations; only genuinely fixed code-coupled vocabularies stay enums.

## DL-015 — Postgres-over-Mongo and Prisma-over-Mongoose.

- **Decision:** Postgres-over-Mongo and Prisma-over-Mongoose.
- **Alternatives considered:** Stay on MongoDB/Mongoose (V1 stack); or Postgres with a query builder / raw SQL.
- **Why this was chosen:** The nine capabilities are relational at their core: multi-axis scoping (year x org x role), referential integrity for RBAC and historized rosters, composite/partial unique constraints, and append-only audit. Postgres gives real FKs, partial/expression/composite uniques, CHECK constraints, triggers, citext, inet, JSONB-where-needed, and BRIN/GIN — none of which Mongo enforces. Prisma gives typed models, migrations, and a single audit choke point via client extensions, and is the idiomatic Next.js 16/React 19 + Neon ORM. Mongoose offers no schema-level integrity for this relational shape.
- **Trade-offs:** Some Postgres power (partial/expression/NULLS-NOT-DISTINCT uniques, deferred-constraint triggers, GIN/BRIN, the lock_guard) lives in raw-SQL migrations invisible to Prisma drift detection and must be hand-maintained and never overwritten by db pull. Pins Neon PG>=15.
- **Future impact:** Integrity is structural rather than app-discipline-dependent — the decisive maintainability factor for a team that turns over every 1-2 years. Raw-SQL objects are catalogued in the maintainer doc.

## DL-016 — JSONB usage policy.

- **Decision:** JSONB usage policy.
- **Alternatives considered:** JSONB-first (snapshot/EAV everywhere) vs typed-columns-first with JSONB confined to schemaless cases.
- **Why this was chosen:** Typed columns for every known shape (all payload tables, lists normalized to child tables). JSONB confined to: page_block_payload.data (genuinely open-ended presentation blocks), audit_log.before/after (heterogeneous write-once snapshots), transition_run.counts (a report blob). This preserves queryability, type-safety, and FK integrity while still giving an escape hatch.
- **Trade-offs:** page_block content is not relationally queryable (acceptable for free-form CMS blocks); a GIN index on data/after supports containment queries where needed.
- **Future impact:** The schema stays self-documenting and diffable as modules grow; EAV's loss of constraints is avoided.

## Session 2 — Database, Prisma, Auth & RBAC decisions

## DL-017 — NextAuth adapter model naming: Prisma models User/Account/VerificationToken mapped to snake_case tables; add app_user.image for adapter compatibility.

- **Decision:** Name the Prisma models `User`, `Account`, `VerificationToken` (so the default `@next-auth/prisma-adapter` resolves `prisma.user/account/verificationToken`) while keeping the snake_case tables `app_user`, `auth_account`, `verification_token` via `@@map`. Add a nullable scalar `image` column to `app_user` (the adapter's OAuth avatar URL), distinct from the curated `avatar_media_id` FK relation.
- **Alternatives considered:** (a) Keep model name `AppUser` and hand-write a custom adapter mapping `prisma.appUser`. (b) Resolve the adapter's `image` solely through `avatar_media_id` (a uuid FK) — impossible, the adapter writes a URL string. (c) Canonical model names + `@@map` + a separate `image` text column.
- **Why this was chosen:** Adopted (c). The default PrismaAdapter is hardcoded to `prisma.user/account/verificationToken`; matching model names avoids a brittle hand-rolled adapter while the tables stay snake_case (SCHEMA_DESIGN intent). The adapter writes `image` (string), so `app_user` needs a real `image` text column; the app resolves the display avatar as `avatar` (media_asset relation) first, falling back to `image`.
- **Trade-offs:** One extra nullable column not in the original SCHEMA_DESIGN dictionary (now documented in the Session-2 addenda there). Model names diverge from the doc's `AppUser` label (table name unchanged).
- **Future impact:** Google + credentials auth work with the canonical adapter out of the box; a curated avatar can later override the OAuth image without schema change.

## DL-018 — audit_log.id as BIGSERIAL (Prisma @default(autoincrement())) rather than GENERATED ALWAYS AS IDENTITY.

- **Decision:** Implement the high-volume `audit_log` PK as `BigInt @id @default(autoincrement())` (emits `BIGSERIAL`).
- **Alternatives considered:** (a) `GENERATED ALWAYS AS IDENTITY` via raw SQL (the SCHEMA_DESIGN note). (b) BIGSERIAL via Prisma autoincrement.
- **Why this was chosen:** Adopted (b). Prisma has no native representation for IDENTITY columns; a raw-SQL `ALTER ... ADD GENERATED ALWAYS AS IDENTITY` would either force every `audit_log.create()` to supply an id (no Prisma default) or create permanent Prisma↔DB drift on `migrate dev`. BIGSERIAL gives the same monotonic bigint PK and is Prisma-native. The append-only guarantee is enforced by the central audit-write service (the only writer) plus the table being insert-only, not by IDENTITY's reject-manual-id property.
- **Trade-offs:** BIGSERIAL permits a manual id insert (IDENTITY ALWAYS would forbid it); acceptable since application code never supplies audit ids.
- **Future impact:** Audit inserts are omittable-id and drift-free; if a stronger guarantee is ever needed it can be added as a controlled migration.

## DL-019 — JWT sessions (24h) with LIVE per-request DB permission resolution; resolves the JWT-revocation open question.

- **Decision:** `session.strategy = 'jwt'`, `maxAge = 24h`. Authorization permissions are resolved LIVE from the DB on every protected action (`lib/rbac/authorize.mjs`), and `requireUser()` re-checks `app_user.status` live. The JWT only carries identity (`uid`) + a coarse `isDeveloper` UI hint.
- **Alternatives considered:** (a) Database sessions for instant server-side revocation. (b) Stuff the permission set into the JWT (fast, but stale until refresh). (c) JWT identity + live DB permission/status checks.
- **Why this was chosen:** Adopted (c). It keeps the simplicity/scalability of stateless JWT while eliminating the usual JWT downside: because permissions and account status are read from the DB per protected request, a revoked role or a suspended account takes effect on the **next request**, not on token refresh. This is the recommended resolution of SCHEMA_DESIGN's "JWT session revocation latency" open question.
- **Trade-offs:** One small DB read per protected action (role_assignment + status). Acceptable and indexed (`role_assignment_active_user_idx`). A stolen-but-valid token remains usable for ≤24h for *authentication* even after sign-out (mitigated by short TTL); all *authorization* is live.
- **Future impact:** No session table to maintain; revocation is effectively immediate for authorization. If instant auth-revocation is ever required, a token denylist or DB sessions can be added.

## DL-020 — Password hashing: argon2id via @node-rs/argon2 (OWASP params).

- **Decision:** Hash passwords with argon2id using `@node-rs/argon2` (m=19456 KiB, t=2, p=1).
- **Alternatives considered:** `node-argon2` (node-gyp native build), `bcrypt`, `scrypt`.
- **Why this was chosen:** `@node-rs/argon2` ships prebuilt binaries (no node-gyp/Vercel build friction), argon2id is the current OWASP first choice, and the chosen parameters are OWASP's recommended baseline. `password_hash` is NULL for OAuth-only accounts.
- **Trade-offs:** A native dependency (prebuilt, low risk). Parameters can be tuned later as hardware improves.
- **Future impact:** Strong, standard credential storage; verify() reads parameters from the encoded hash so params can evolve without breaking existing hashes.

## DL-021 — Roster cardinality: denormalized appointment.is_singleton + partial unique (singletons) AND deferred count trigger (multi-holder).

- **Decision:** Keep DL-009's two-guard design. Because a partial-index predicate cannot read `position.max_holders`, denormalize `appointment.is_singleton` (maintained by the `appointment_type_guard` trigger from `position.max_holders`) and build the concurrency-safe singleton guard as `appointment_singleton_position_uq (academic_year_id, org_unit_id, position_id) WHERE archived_at IS NULL AND status <> 'archived' AND is_singleton`. Keep the deferred `appointment_cardinality_guard` count trigger for bounded multi-holder positions (max_holders > 1).
- **Alternatives considered:** (a) Trigger-only for all cardinality (the initial Session-2 build) — has a TOCTOU/MVCC race under READ COMMITTED: two concurrent inserts of the same singleton can both pass the count check and commit. (b) `pg_advisory_xact_lock` in the trigger. (c) Denormalized flag + partial unique + count trigger.
- **Why this was chosen:** Adopted (c) after the Session-2 adversarial review flagged the race. A real partial UNIQUE index serializes concurrent inserts at the index level regardless of isolation, which is exactly why DL-009 mandated it; the count trigger alone cannot. The denormalized boolean is the only way to express the singleton predicate in a partial index.
- **Trade-offs:** One extra boolean column maintained by a trigger. Negligible.
- **Future impact:** "single secretary/PIC/warden, multiple coordinators" is guaranteed even under concurrent writes.

## DL-022 — Add the org_unit parent-hierarchy guard trigger (same-year + allowed-child-type).

- **Decision:** Add `org_unit_hierarchy_guard` (BEFORE INSERT/UPDATE on org_unit): when `parent_id` is set, the parent must be in the SAME academic year and `(parent_type, child_type)` must exist in `org_unit_type_allowed_child` (also blocks self-parenting).
- **Alternatives considered:** App-layer-only enforcement; or the plain single-column FK alone (the initial Session-2 build).
- **Why this was chosen:** SCHEMA_DESIGN mandates "same-year + allowed-child-type trigger-checked" for `org_unit.parent_id`; the initial build shipped only a single-column FK, leaving cross-year hierarchy contamination and disallowed parent types possible — the same class of bug DL-008's composite FK closed for appointments. The Session-2 review caught the omission.
- **Trade-offs:** A trigger (more moving parts) vs an unenforceable rule. Worth it for the structural backbone.
- **Future impact:** The org hierarchy cannot be corrupted across years or against the allowed-child rules even by direct DB writes; RBAC scope traversal over the hierarchy stays sound.

## DL-023 — ESM `.mjs` for all new server/library modules (no package-wide "type":"module").

- **Decision:** Author Prisma client, auth, RBAC, CMS, seed, and tests as `.mjs` ES modules; do NOT set `"type":"module"` in package.json.
- **Alternatives considered:** Set `"type":"module"` globally; or keep `.js` with CommonJS.
- **Why this was chosen:** `empty-module.js` (referenced by `next.config.mjs`) uses CommonJS `module.exports`, so a package-wide `"type":"module"` would break it. `.mjs` files are unambiguously ESM for node (seed), Vitest, and the Next bundler alike, with no per-file config.
- **Trade-offs:** Explicit `.mjs` extensions in imports; mild inconsistency with V1 `.js` app files (which Next transpiles as ESM regardless).
- **Future impact:** Seed/tests run under plain `node`/Vitest; app code imports the same modules through the bundler. A future migration to `"type":"module"` (after `empty-module.js` is removed) is optional.

## DL-024 — Interim auth gating of legacy V1 surfaces (events API now permission-gated; admin page gated on isDeveloper) pending their rebuilds.

- **Decision:** Gate the legacy `POST /api/events` with `requirePermission('content.create')` now (closing KNOWN_ISSUES #2), and change `app/admin/page.js` from the dead V1 `session.user.isAdmin` flag to `session.user.isDeveloper`. Full rebuilds remain scheduled: events on Postgres in Session 6, the RBAC-gated admin panel in Session 9.
- **Alternatives considered:** Leave the V1 handlers untouched until their rebuild sessions; or fully rebuild now (out of Session-2 scope).
- **Why this was chosen:** KNOWN_ISSUES #2 (unauthenticated event writes) is explicitly assigned to Session 2, and the review confirmed the endpoint was an open write hole. Gating it with the new utility both closes the hole and demonstrates the authorization utility "used by a protected handler" (task #7), without prematurely doing the Session-6 rebuild. The admin page referenced a flag the new session never sets (fail-closed); pointing it at the real `isDeveloper` flag makes it usable in the interim while the real boundary stays server-side.
- **Trade-offs:** The events handler still writes to Mongo until Session 6; the admin UI gate is client-side (UX only). Both are documented interim states.
- **Future impact:** No unauthenticated mutation endpoint remains. Sessions 6/9 replace these with Postgres-backed, server-enforced implementations.

## DL-025 — Defer the central audit-write Prisma extension to Session 3 (CMS Foundation).

- **Decision:** Create the `audit_log` table + indexes now, but implement the single central audit-write Prisma Client extension / service in Session 3, when the first audited mutations (CMS create/update/publish/restore) actually exist. This supersedes DL-012's implicit Session-2 placement for the *writer* only (the *schema* is delivered in Session 2).
- **Alternatives considered:** Build the extension now with nothing to audit; or leave it silently unscheduled.
- **Why this was chosen:** An audit writer with no mutating call sites would be dead, untestable code. Session 3 introduces the mutation pipeline it must wrap, so the choke point is built and tested against real operations there. TODO.md already schedules it under Session 3; this records the rationale so the deferral is explicit, not silent.
- **Trade-offs:** Capability-8 enforcement lands one session later than the schema. No runtime risk (no audited mutations exist yet in the new pipeline).
- **Future impact:** The audit extension is implemented once, against real CMS mutations, with coverage tests — exactly where DL-012 intended a single choke point.

## DL-026 — Confirmed defaults for the SCHEMA_DESIGN "Open questions".

(2) Locked-year errata = controlled unlock→correct→relock OR append a new content_revision; `lock_guard` blocks INSERT/UPDATE/DELETE on org_unit/appointment/content_item and UPDATE/DELETE on content_revision in a locked year (revision INSERT allowed for the append path). (4) Approval workflow: single-state `review` for now; no reviewer-assignment table until a multi-step sign-off is required. - **Why this was chosen:** Each default is the least-surprising, lowest-complexity choice that satisfies the current single-institute requirements while leaving the heavier option additive.
- **Trade-offs:** Some conveniences (multi-step review, year-namespaced URLs, term scoping) are deferred; all are additive later.
- **Future impact:** Future sessions implement against these confirmed defaults; revisiting any is an additive change, not a rework.

## DL-027 — Migration delivery: single hand-assembled init migration; evolve the dev DB non-destructively (no `migrate reset`).

- **Decision:** Deliver one init migration = Prisma-generated base DDL (`migrate diff --from-empty`) + a hand-written raw-SQL tail (extension, partial/expression/NULLS-NOT-DISTINCT uniques, GIN/BRIN, CHECKs, triggers), applied via `prisma migrate deploy`. During the Session-2 review fixes, the live dev DB was evolved by applying a non-destructive delta via `prisma db execute` + re-stamping the init checksum (verified by `prisma migrate status`), rather than `prisma migrate reset` (which the Prisma AI-safety guard blocks without explicit user consent, and which destroys data).
- **Alternatives considered:** (a) `migrate dev` (needs a Neon shadow DB / CREATEDB the pooled user lacks). (b) `migrate reset` to re-apply the corrected init (destructive; blocked by the AI guard). (c) Offline `migrate diff` to author the SQL + `migrate deploy`, evolving the existing DB with a checked delta.
- **Why this was chosen:** Adopted (c). It avoids the Neon shadow-DB friction, never runs a destructive op, and keeps a single clean init migration that a fresh `migrate deploy` reproduces exactly. The checksum re-stamp keeps the recorded migration consistent with the corrected file so future `migrate` commands don't flag drift.
- **Trade-offs:** The raw-SQL objects are invisible to Prisma drift detection and must be hand-maintained (catalogued in SCHEMA_DESIGN + covered by `tests/migration.test.mjs`). Never run `prisma db pull` (it would drop them from the schema's view).
- **Future impact:** Future sessions add forward migrations via the same diff+deploy pattern; raw-SQL objects evolve in explicit, reviewed migrations.

## Session 3 — CMS Foundation decisions

## DL-028 — Central audit-write as a two-path Prisma Client `$extends` (auto per-statement + semantic per-operation), recursion-safe via the base client, best-effort and after-commit.

- **Decision:** Implement capability-8 / DL-012 as ONE choke point in `lib/cms/audit.mjs`, mounted in `lib/prisma.mjs`, with two complementary paths: (1) an **auto** `$extends` query extension that audits every mutating op (create/update/delete/upsert + `*Many`) on audited models — the safety net + the freebie for future single-statement modules; (2) a **semantic** path used by the CMS service: each multi-step business operation runs inside one transaction with `suppressAuto` set (via an `AsyncLocalStorage` actor context), then writes exactly ONE high-level `audit_log` row (action = publish/unpublish/archive/restore/grant_role/…) AFTER the transaction commits. The audit row and all before-reads use the **un-extended `base` client**, so an audit write never re-enters the extension (recursion-safe). Auditing is **best-effort**: a failed audit write is logged and never rolls back / masks the user's mutation. Actor + ip/user-agent come from the context; explicit fields win.
- **Alternatives considered:** (a) Auto-only per-statement audit — produces 3–5 low-level rows per business action (publish touches 3 tables), which is the wrong grain for a human-readable activity log whose enum is already operation-level. (b) Semantic-only — loses the automatic "future modules audited for free" promise of DL-012. (c) Writing the semantic row INSIDE the transaction (atomic) — rejected for now because a failed/contended audit insert must never roll back a user's content change; the after-commit window is a deliberate trade.
- **Why chosen:** The `audit_action` enum is operation-level, so one semantic row per business action is the right grain; the auto path still gives uniform, automatic coverage for everything else and for future modules. Using `base` for the audit write is the clean way to avoid the classic extension-recursion trap. `suppressAuto` prevents double-counting between the two paths.
- **Trade-offs:** (1) The semantic row is written after commit, so a process crash between COMMIT and the audit insert yields a committed-but-unaudited mutation (one lost row, no integrity impact). (2) The auto path writes on `base`, so if a future caller wraps an auto-audited single-statement write in their own interactive `$transaction` that then rolls back, the audit row would persist (audit-on-rollback). Both are documented (KNOWN_ISSUES #24); atomic multi-step flows use the semantic path and never rely on auto-audit. (3) Auto update/delete costs one extra before-read.
- **Future impact:** New modules are audited automatically; if stronger guarantees are ever needed, the semantic `recordAudit` already accepts a `tx` and can be moved inside the transaction. `Account`/`User`/`VerificationToken` are skipped (auth churn noise); user/role admin actions will be audited semantically by their Session-9 service.

## DL-029 — Honor DB guards; never re-implement them in app code. Translate violations to friendly HTTP-shaped errors via `mapDbError`.

- **Decision:** The CMS service relies entirely on the DB's integrity guards (the one-draft/one-published partial uniques, the `content_item_pointer_guard`, `lock_guard`, and payload CHECK constraints) and honors their required write ORDERING (e.g. publish supersedes the prior published revision before marking the draft published). It does NOT duplicate those rules in JavaScript. When a guard rejects a write, `lib/cms/errors.mjs#mapDbError` maps the Postgres/Prisma error (matched by trigger/constraint name in `err.message`/`err.meta`, plus Prisma codes P2002/P2003/P2025) into a `CmsError` with a stable `code` + HTTP `status` (e.g. `YEAR_LOCKED` 409, `ONE_DRAFT`/`ONE_PUBLISHED` 409, `SLUG_TAKEN` 409, `PUBLISH_WINDOW`/`INVALID_CAPACITY`/`INVALID_MEAL_TIME` 422, `POINTER_GUARD` 500). App-side validation is limited to caller-input concerns (required payload fields → `CmsValidationError` 422, unknown content type, missing year/org binding) that are cheaper to catch before the DB round-trip.
- **Alternatives considered:** (a) Re-check the invariants in app code before writing (TOCTOU-racy, duplicates the DB, drifts). (b) Let raw Prisma errors bubble to callers (leaks internals; no stable contract). 
- **Why chosen:** Single source of truth for integrity (the DB) with a thin, testable translation layer is the most maintainable for rotating students and matches the schema's "structural, not app-discipline" philosophy (DL-015).
- **Trade-offs:** `mapDbError` matches on guard/constraint NAMES, so renaming a constraint without updating the matcher would silently fall through to a generic 500 — mitigated by unit tests over the matcher strings and a live-DB test that provokes a real `YEAR_LOCKED` through the actual trigger.
- **Future impact:** Every future mutating surface wraps its DB block in `withMappedDbErrors` and gets consistent friendly errors for free.

## DL-030 — Content lifecycle & restore semantics; interactive-transaction timeout ceiling for Neon.

- **Decision:** (1) `editDraft` edits the open draft in place when one exists; if the item was published with no open draft, it auto-opens a NEW draft revision cloned from the published (or latest) revision and applies the edits — the live published revision is never touched. (2) `publish` supersedes the prior published revision, marks the draft published, repoints `published_revision_id`, and clears `draft_revision_id`. (3) `unpublish` returns the formerly-published revision to an editable draft when none is open, else supersedes it. (4) `restore` OVERWRITES the open draft in place (reusing that row, honoring the one-open-draft partial unique) and records `is_restore_of_revision_id`; it creates a new draft only when none is open. (5) Every mutating op reuses the Session-2 RBAC util (`assertPermission`) against the item's (year, org-lineage) scope and authorizes BEFORE any state-based precondition (no auth-after-disclosure); a `system` actor bypasses for seeds/migrations. (6) CMS interactive transactions are given a generous `{ timeout: 30000, maxWait: 20000 }` ceiling so a cold/remote Neon compute (observed ~1.5–2.5s per round-trip) does not trip Prisma's default 5s limit (P2028); warm in-region these finish in well under a second, so the ceiling never trips in normal serverless operation.
- **Alternatives considered:** restore as a NEW revision every time (would accumulate draft rows and fight the one-open-draft unique); editing the published revision directly (breaks draft/publish isolation); default tx timeout (fails on cold Neon).
- **Why chosen:** Matches SCHEMA_DESIGN capability 6/7 ("admins edit only the draft", "restore overwrites the open draft in place"); the generous tx ceiling is a safety value, not a behavior change.
- **Trade-offs:** Restore is destructive to the current draft's contents (by design — it's a restore). The tx ceiling masks pathological slowness up to 30s; acceptable for a safety bound.
- **Future impact:** Session-9 admin UI and Session-6 events/announcements build directly on these service functions; the public read path is `lib/cms/visibility.mjs`.

## Session 4 — Academic Year Engine decisions

## DL-031 — The Transition Wizard copies as a sequence of idempotent statements (NOT one interactive transaction); crash-safety comes from idempotence + a `transition_run` status, not atomicity.

- **Decision:** `runTransition` (`lib/year/transition.mjs`) performs the structure / appointment / content / role-assignment copy as a sequence of ordinary statements with **auto-audit suppressed**, recording exactly ONE semantic `audit_log` row (action `transition`) after the run completes. It is NOT wrapped in a single `prisma.$transaction`. Idempotence is structural: each phase skips target rows that already exist — org_units by `(academic_year_id, lineage_key)` (the unique), content_items by `(content_type, lineage_key)`, appointments by `(org_unit, position, person)` (+ a singleton-slot pre-skip), role grants by `(user, role, lineage)`. A completed run for a `(source, target)` pair makes a plain re-run a no-op (honoring `transition_run_one_completed_uq`); `{ force: true }` re-opens and re-syncs **the same** completed run row (never a second 'completed') and, on failure, **restores** that row's prior status/counts so a successful run's provenance is never destroyed. The org-unit parent wiring reconciles **all** source units with a target twin (not only freshly-created ones) so a partial/crashed prior run self-heals on resume; per-content-item cloning (item → revision → payload → draft pointer) is wrapped in a **small per-item `$transaction`** so a mid-item failure rolls back the bare header rather than leaving an unrepairable, forever-skipped row.
- **Alternatives considered:** (a) One big interactive `$transaction` for the whole wizard — over high-latency Neon a ~40-unit + content copy can exceed any sane tx ceiling (P2028), and a single failure discards all progress. (b) No idempotence, rely on the operator not re-running — fragile and unsafe. (c) Non-transactional but with the header row alone as the skip marker — the original Session-4 build; the adversarial review showed a mid-item crash then leaves a half-created org_unit/content_item that is skipped forever (silent hierarchy corruption / revision-less content).
- **Why chosen:** Matches the schema's "nothing overwritten, re-runnable" intent (SCHEMA_DESIGN cap 1) and the Neon latency reality. Statement-level idempotence + a status-tracked `transition_run` gives safe resume without a giant transaction; the two targeted `$transaction`s (per content item) and the all-units parent reconciliation close the only windows where "row exists" did not mean "row is complete."
- **Trade-offs:** The whole run is not atomic — an interrupted run can leave the target year partially populated, but a re-run safely completes it (and reports accurate per-entity `counts`). A forced re-sync with a *changed singleton holder* pre-skips the new holder (leaving the prior target appointment for manual reconciliation) rather than aborting the batch.
- **Future impact:** Session 5 (org model) and Session 9 (admin wizard UI) drive `runTransition` directly; the counts/status contract and idempotence guarantees are stable. If true atomicity is ever required, the phases already use friendly-error mapping and could be moved under one transaction behind a feature flag.

## DL-032 — Public year selector: the live publish window is enforced only for the CURRENT year; a selected PAST year is an archive that shows all its published content.

- **Decision:** `lib/year/public.mjs#listPublicContentForYear` / `getPublicItemBySlugForYear` apply the event/announcement `publish_from..publish_until` window **only when the selected year is the current year** (identical to `lib/cms/visibility.mjs`). For a past (active-but-not-current, or locked) year the window is ignored, so that year's published events/announcements stay browsable forever. Callers may override via `applyWindow`. Only `active`/`locked` years are publicly selectable; `planning` years are never exposed.
- **Alternatives considered:** (a) Always enforce the window for the selected year — hides every past event whose window closed, making the archive empty and useless. (b) Never enforce the window — would leak not-yet-live current-year events. (c) Year-conditional enforcement (chosen).
- **Why chosen:** The publish window is a "what is live right now" gate that is meaningful only for the live year; historically it is noise. Year-conditional enforcement keeps current-year behavior byte-for-byte identical to Session 3 while giving a sensible archive.
- **Trade-offs:** A past-year archive shows events that were only ever scheduled (published with a future-then-past window). Acceptable for an archive; `applyWindow:true` is available if a caller wants strict windowing.
- **Future impact:** Session 9 public pages get a year picker (`listSelectableYears`) and per-year archives for free; routing/URL year-namespacing remains a separate concern (DL-026).

## DL-033 — Promote the post-commit "audited mutation" wrapper (+ Neon tx ceiling) to one shared module used by both the CMS service and the Year engine.

- **Decision:** Extract `auditedMutation` and `TX_OPTS` into `lib/cms/audited-mutation.mjs`; `lib/cms/content.mjs` and `lib/year/context.mjs` (re-exported to `lock.mjs`) both import them. One place now owns the "run in a transaction with auto-audit suppressed, then write exactly one semantic `audit_log` row after commit via the un-extended `prismaBase`" pattern (DL-012 / DL-028).
- **Alternatives considered:** (a) Copy the 12-line wrapper into each module (the original Session-4 build) — the adversarial review flagged the two copies as drift-prone (a fix to the suppress/record ordering would have to be made twice). (b) A class/base-service abstraction — over-engineered for two call sites.
- **Why chosen:** Single source of truth for the audit-write grain with zero behavior change; the live CMS + Year suites both exercise it.
- **Trade-offs:** One more small module in the import graph.
- **Future impact:** Every future audited mutating service (Session 5 org, Session 6 events, Session 9 admin) imports the one wrapper and gets uniform semantic auditing.

## Session 5 — Organization Model decisions

## DL-034 — Person identity is the cleaned full name (case-insensitive); V1 role-mailbox "emails" are NOT migrated to `person.email`.

- **Decision:** A `person` is deduplicated by its whitespace-cleaned full name, compared **case-insensitively** (`lib/org/people.mjs#upsertPerson` uses `fullName { equals, mode:'insensitive' }`). The institutional "emails" attached to people in the V1 hostel data (e.g. `warden.egret@iitjammu.ac.in`, `hsec.boys@iitjammu.ac.in`) are **shared ROLE mailboxes**, not personal identity emails, so they are NOT written to `person.email` (which is `UNIQUE WHERE NOT NULL`). The hostel's office mailbox is kept on `hostel_profile_payload.office_email` instead. `upsertPerson` additionally drops any `email` that would collide with a different existing person (defensive). The few V1 spelling variants of one human ("Mr. Sumit Raj Ghosh" vs "Sumit Raj Ghosh"; "Akash Awale" vs "Dr. Akash Subhash Awale") are reconciled to a single canonical name **in the dataset** (`lib/org/data/*`), not by a fuzzy matcher.
- **Alternatives considered:** (a) Key by email — impossible: the same warden mailbox is reused by two different wardens, and `person.email` is UNIQUE, so it both mis-merges distinct people and would throw P2002. (b) Migrate the role mailboxes onto `person.email` first-wins — silently drops real contacts and still mis-attributes. (c) A fuzzy / honorific-and-middle-name normalizer for the dedup key — over-engineered and risky (could merge genuinely different people); the divergent cases are a tiny, known set better fixed as data.
- **Why chosen:** Names are the real identity in this single-institute directory (the same "Mehul Gupta" is hostel secretary across four hostels = one person, four appointments — exactly the normalization win). Case-insensitivity matches the intent of the pure `personKey` helper without needing a stored generated column. Reconciling the 2 known variants as data keeps the dedup logic simple and deterministic, and keeps `inferPersonType` order-independent (the honorific is now consistent across a person's occurrences).
- **Trade-offs:** Two genuinely different humans who share an exact cleaned name would merge (acceptable at this scale; none in the V1 data). Personal contact emails are not captured for migrated people (the role mailbox lives on the unit profile); a future admin can add identity emails per person.
- **Future impact:** The directory stays clean and FK-safe; Session 9's people admin can add/curate emails and optionally link `app_user` accounts (then `person_email_link_guard` enforces email agreement).

## DL-035 — The campus-wide mess committee is attached to the first mess unit.

- **Decision:** V1 models the mess committee as ONE campus-wide roster (a single list, not per-mess). Because the mess position definitions (`mess_secretary`, `mess_committee_member`) require a mess-type `org_unit` (`appointment_type_guard`), the importer attaches the entire committee to the **first mess unit by sort order** ("Annapurna Mess (2nd Floor)") as the canonical committee roster, preserving each V1 title in `appointment.title_override` ("AD - Mess Management", "Mess Warden - Canary", …). "Mess Secretary" maps to the singleton `mess_secretary`; everyone else to the unlimited `mess_committee_member`.
- **Alternatives considered:** (a) Duplicate the whole committee onto all 5 messes — 16×5 fabricated appointments and a singleton-secretary contradiction. (b) Best-effort per-mess attribution by parsing titles ("Mess Warden - Canary" → Canary) — unreliable and incomplete (most members carry no mess). (c) A new institute-level "mess administration" unit type — schema/seed change unjustified for V2.
- **Why chosen:** It is faithful to "one committee", keeps cardinality sane (one secretary, many members), and satisfies the type guard with zero schema change. `title_override` preserves the human-meaningful V1 labels.
- **Trade-offs:** The committee is rendered under one mess rather than as a standalone institute roster; per-mess attribution is deferred until the data distinguishes it.
- **Future impact:** If a future year provides per-mess committees, the same services attach them to the right unit with no code change.

## DL-036 — Fix `appointment_type_guard.is_singleton` via a forward migration (`COALESCE`), not by editing the init migration.

- **Decision:** The Session-2 `appointment_type_guard` trigger computed `NEW.is_singleton := (v_max_holders = 1)`, which evaluates to **NULL** when `position.max_holders IS NULL` (unlimited positions: coordinators, caretakers, wellness wardens, attendants, mess committee members), violating `appointment.is_singleton NOT NULL`. The bug was latent because no prior session created an unlimited-holder appointment (Session 4 only copied a singleton secretary); the Session-5 live test (`tests/org.db.test.mjs`) surfaced it on the first coordinator insert. Fixed with `NEW.is_singleton := COALESCE(v_max_holders = 1, false)` in a NEW forward migration (`20260628130000_fix_appointment_singleton_guard`, a `CREATE OR REPLACE FUNCTION` — idempotent, leaves the trigger binding intact), applied to Neon via `prisma migrate deploy`. The recorded init migration is left untouched (DL-027: never rewrite/`db pull`/`migrate reset`).
- **Alternatives considered:** (a) Edit the init migration's function in place + re-stamp the checksum — breaks the "init is immutable once applied" invariant and risks drift. (b) A column default / app-side default for `is_singleton` — the trigger overwrites it, so the NULL would still land. (c) Make `is_singleton` nullable — weakens the singleton partial-unique predicate.
- **Why chosen:** A forward `CREATE OR REPLACE` migration is the sanctioned evolution path (DL-027), is safe to re-apply, and a fresh `migrate deploy` from empty reproduces the corrected behavior (init creates the buggy function, the fix migration corrects it, in order).
- **Trade-offs:** One more migration directory; the raw-SQL trigger remains invisible to Prisma drift detection (catalogued as before).
- **Future impact:** Establishes the pattern for future raw-SQL trigger fixes — a new `CREATE OR REPLACE` migration, never an init rewrite. The live-DB test suite is what guards these triggers against regression.

## DL-037 — Events & Announcements are CMS `content_type` CALLERS (no new pipeline); the V1 Mongo events API is REPLACED by a CMS-backed route and Mongoose is retired from the request path.

(b) Keep the Mongo events collection behind the gated V1 endpoint — leaves two databases, the base64 bloat (#5), and no edit/delete (#16). (c) Retire `/api/events` entirely and only render via Server Components — loses a usable authenticated create path before the Session-9 admin panel.

Driving them through the one CMS service gives draft/publish, version history, audit, year-scope and the publish-window guard with zero new integrity code (DL-029) and closes #2/#5/#9/#16 structurally. A CMS-backed `/api/events` keeps a working, validated, authenticated event-create path now (used by the interim admin form) while the rich admin UI lands in Session 9.

The V1 `/admin` form had to be updated (base64 file upload → image-URL field) since the new route rejects base64.
- **Future impact:** Resources (Session 7), the admin panel (Session 9) and any future news-like module reuse the same CMS service + this read/shape pattern. Event registration/RSVP, if ever needed, is a new capability-9 module, not a change here.

## DL-039 — V1 base64 event images become lightweight `media_asset` PLACEHOLDER rows, never inline blobs (KNOWN_ISSUES #5).

(b) Accept base64 in the API and store it — same bloat + no validation. (c) Decode/upload to Cloudinary in Session 6 — that is the Session-7 Media tool's job; doing it here duplicates it.

- **Future impact:** Session 7's media migration upgrades placeholders/`/public` paths to Cloudinary assets; the public read layer already resolves `coverMediaId` → URL, so covers light up with no page change.

## DL-040 — Public (anonymous) event/announcement reads are gated to `audience='public'`; the audience scope is enforced, not just displayed.

- **Decision:** `audience_type` (public|students|faculty|staff|internal) is treated as an access scope, not a cosmetic label. The public read layer (`lib/events/public.mjs`) filters to `PUBLIC_AUDIENCES = ['public']` by default via the pure, tested `filterByAudience`, so content an editor marked non-public is never surfaced to anonymous visitors. Callers may widen the allowed set through an `audiences` option (the seam a future role-aware / authenticated view uses). The write path (`POST /api/events`) validates `audience` against the enum and returns a friendly 422 for unknown values.
- **Alternatives considered:** (a) Show all audiences publicly and treat `audience` as a tag (the initial Session-6 build) — the adversarial review flagged it as an information-disclosure path: an `internal`/`staff` announcement would render on the anonymous page (even with a "For Internal" badge). (b) Drop the column from the public surface entirely — loses a real, modeled capability. (c) Filter at the SQL `where` — `audience` lives on the payload (event/announcement_payload), not on `content_item`, so it is filtered in the shaping layer after the payload is loaded.
- **Why chosen:** The least-surprising, safe default for an anonymous visitor is "public only"; a senior engineer would not ship a path that puts editor-marked-internal content on a public page. Keeping it a parameter leaves role-aware widening additive for Session 9.
- **Trade-offs:** `students`/`faculty`/`staff`/`internal` content is hidden from the anonymous public pages until a role-aware authenticated view passes a wider `audiences` set (Session 9). The documented public-visibility rule (SCHEMA_DESIGN) is extended by this audience clause for windowed news types.
- **Future impact:** Session 9's authenticated/admin views pass the viewer's allowed audiences; the gate is one pure function with full unit + live coverage.

## DL-041 — Resources are CMS `content_type='resource'` CALLERS (DL-037 pattern), each with its OWN content lineage; campus-wide infra PDFs bind to the first unit of a kind (DL-035 pattern).

Campus-wide infra PDFs render under one unit, not as a standalone index (the Session-9 admin/index page can group them via `listPublicResourcesByUnit`).

## DL-042 — Media: a thin service layer over `media_asset` with a SINGLE delivery-URL resolver; curated ops audited via `auditedMutation`, bulk inventory via `prismaBase`; the Cloudinary uploader is INJECTED.

- **Decision:** `lib/media/` splits into (1) `cloudinary.mjs` — PURE helpers (`cloudinaryUrl`, `publicIdFromPath`, `signUploadParams`, `resolveDeliveryUrl`, env config) + the one impure `uploadFileToCloudinary`; (2) `service.mjs` — curated `media_asset` CRUD (`createMediaAsset`/`updateMediaAsset`/`archiveMediaAsset`) through the shared `auditedMutation` (exactly one semantic audit row; authorize `media.upload`/`media.update`/`media.delete` FIRST), reads (`listMediaAssets`/`getMediaAsset`) + `shapeAsset`, and the bulk, audit-bypassing `findOrCreateInventoryAsset` (default `prismaBase`); (3) `migrate.mjs` — the migration tool. `resolveDeliveryUrl(asset)` is the ONE place that turns a row into a public URL (migrated→Cloudinary URL, else stored url); for a transformed PDF it injects `format: 'pdf'` so Cloudinary returns the file, not a rasterized page. `updateMediaAsset` only patches curate-able metadata (altText/kind/mimeType/width/height) — storage/cloudinary fields are migration-owned. The real Cloudinary uploader is an **injected** dependency so tests use a deterministic fake (no network/credentials).
- **Alternatives considered:** (a) Audit curated media writes only via the auto `$extends` path — produces a low-level CRUD row, not the operation-level grain; the semantic path matches every other service. (b) Hard-wire the real uploader — untestable without Cloudinary creds and a network. (c) Scatter delivery-URL building at call sites — drift; one resolver keeps a future CDN/transform change in one file.
- **Why chosen:** Mirrors the established service pattern (CMS/org/events): authorize-then-audited-mutation for curated single writes, `prismaBase` for bulk inventory (the importers would otherwise flood `audit_log`). The injected uploader is what makes the migration tool live-testable. The PDF `format` fix prevents a (latent) broken transformed PDF URL.
- **Trade-offs:** Two write paths (curated audited vs bulk bypass) the caller must choose correctly — documented in the service header. `media.read` is global scope (media is institute-wide, not org-bound).
- **Future impact:** Session 9's media library admin builds on `listMediaAssets`/`createMediaAsset`/`updateMediaAsset`/`archiveMediaAsset`; any new module resolves URLs through `resolveDeliveryUrl` and registers inventory via `findOrCreateInventoryAsset`.

## DL-043 — The Admin Media Migration Tool: idempotent, reversible, dry-run-first `/public` → Cloudinary, reconciling the base64 placeholders; bulk writes bypass audit with ONE summary row.

- **Decision:** `lib/media/migrate.mjs#migratePublicAssets` migrates the `media_asset` inventory rows the importers created (`storage_provider='local'`, `original_path='/…'`) to Cloudinary: it derives a deterministic `public_id` from the path, uploads (injected uploader), and repoints the row (`cloudinary_public_id`/`url`/`migrated_at`). **Idempotent** — the candidate query excludes rows that already carry a `cloudinary_public_id`, so a re-run migrates 0. **Reversible** — `original_path` is never cleared, and `rollbackMigration` restores `storage_provider='local'` + `url ← original_path` and clears the cloudinary fields (itself idempotent; it does NOT delete the remote asset — the deterministic `public_id` means a re-migrate reuses it). **Dry-run** computes the full plan with zero writes (the CLI default). It also reconciles the Session-6 base64 placeholders (`BASE64_PLACEHOLDER_URL`, DL-039): without a `base64Resolver` they are reported `base64Pending` (bytes live in the Session-1 backup); with one they upload + repoint. All per-row writes use `prismaBase` (no audit flood over ~105 assets); ONE semantic `audit_log` row summarizes the run. A `filter` option scopes a run (a folder, or a test's own rows). `media.migrate` is authorized first.
- **Alternatives considered:** (a) Store the Cloudinary URL but drop `original_path` — non-reversible. (b) One big transaction over all assets — Neon-latency-hostile (DL-031) and not resumable. (c) Audit every per-asset write — floods `audit_log`; the run-summary row is the right grain. (d) Hard-delete the `/public` files as part of the migration — destructive and breaks rollback; deferred to a manual post-verify prune (KNOWN_ISSUES #18).
- **Why chosen:** Idempotent + reversible + dry-run is the mandated contract (NEXT_TASK) and the safe operator UX (`db:migrate:media` previews; `-- --apply` mutates; `-- --rollback` reverses). Statement-level idempotence (not a giant tx) matches the importers (DL-031). Keeping `original_path` is what makes rollback possible and a re-migrate stable.
- **Trade-offs:** Rollback leaves the (now-orphan) Cloudinary asset in place (manual cleanup; harmless — re-migrate reuses the same `public_id`). `/public` is not auto-pruned, so #18's ~74 MB only shrinks after the operator runs `--apply` and verifies, then deletes the files out-of-band. base64 placeholders need an out-of-band resolver (their bytes are only in the backup).
- **Future impact:** Session 8's developer console can surface migration status/counts; Session 10 prunes `/public` after a verified migration. The tool is the template for any future bulk asset move.

## Session 8 — Developer Console decisions

## DL-046 — The Developer Console is a thin READ-MOSTLY CALLER layer (`lib/devconsole/*`) over the Session 2–7 plumbing; it adds NO new audit writer / mutation / rollback pipeline.

- **Decision:** Session 8 is delivered as `lib/devconsole/{authorize,audit,status,reports,backups}.mjs` + two gated routes (`app/api/dev/{status,audit}/route.js`) + a CLI (`scripts/devconsole.mjs`, `npm run db:console`). It is overwhelmingly a READER. (1) The **audit-log viewer** (`audit.mjs`) reads `audit_log` (filter by actor/entity/action/year/time-range; newest-first keyset pagination via the monotonic BIGSERIAL `id`, DL-018; stats via `groupBy`) — it adds NO writer (the central `$extends` writer of DL-028 is the only one). (2) **Monitoring** (`status.mjs`) is read-only: a DB-connectivity+latency probe, a `prisma migrate status`-shaped diff of the on-disk migrations vs the `_prisma_migrations` ledger, `transition_run` history (reusing `lib/year/transition.mjs#listTransitionRuns`), and the media-migration plan. (3) **Reports** (`reports.mjs`) parse `docs/Token_Usage.md` + read live infra sizes for an indicative cost view. (4) The only WRITES are the `backup_record` ledger (`recordBackup`/`markBackupVerified`) — done through the **shared `auditedMutation`** (one semantic row each, like every other service) — and the **recovery actions**, which are THIN DELEGATES: `rollbackMediaMigration` → `lib/media/migrate.mjs#rollbackMigration` (DL-043), `forceTransitionResync` → `lib/year/transition.mjs#runTransition({force:true})` (DL-031). Authorization is `authorizeConsole(actor, keys)` — an **any-of** gate (additive-union RBAC, developer/`grants_all` short-circuit, `{system:true}` bypass) mirroring `lib/media/migrate.mjs#authorizeMigrate`.
- **Alternatives considered:** (a) A bespoke ops datastore / a second audit writer for console events — duplicates the DL-028 choke point and the schema's `backup_record`/`transition_run` tables. (b) Re-implementing rollback / transition logic inside the console — duplicates the tested, idempotent services and their guards. (c) Building the full console UI now — that is the Session-9 admin panel; Session 8 delivers the read/service/route layer it will render.
- **Why chosen:** The schema already shipped `audit_log` (cap 8), `transition_run` (cap 1) and `backup_record` (Session-2). Session 8's job is to *consume* that plumbing, not extend it — so the console is a caller layer with one new permission gate, honoring the "reads more than it writes / no new pipeline" guard rail. Delegating recovery keeps a single source of truth for every mutation.
- **Trade-offs:** The status sub-helpers (`checkDatabase`/`getMigrationStatus`/`getTransitionStatus`/`getMediaMigrationStatus`/`getInfraUsage`) are individually UNGATED building blocks (the gated public surface is the `getSystemStatus`/`getDevConsoleReports` aggregators + the routes); this is deliberate so the CLI/tests can call them directly and the media plan needs no `media.migrate`. A recovery delegate requires the caller to satisfy BOTH the console gate AND the underlying service's own gate (defense-in-depth), which a developer/super_admin satisfies via `grants_all`.
- **Future impact:** The Session-9 admin panel renders these readers as a UI (status dashboard, audit viewer, backup ledger, recovery buttons) with zero new backend; any future module is observable for free (its audit rows show in the viewer, its migrations in the status diff).

## DL-047 — The audit viewer is gated on the dedicated `audit.read` (not the broad `dev.console`), and bulk list/timeline rows DATA-MINIMIZE PII (ip / user-agent / before-after only in the single-entry detail view).

- **Decision:** `lib/devconsole/audit.mjs` authorizes every read on `audit.read` ALONE (not `audit.read` OR `dev.console`), and `/api/dev/audit` gates the boundary on the same single key — so the boundary and the service authorize identically. `shapeAuditEntry` includes the core who/what/when on every row (incl. the actor's email — that IS the audit subject) but emits the incidental per-row `ip_address` / `user_agent` and the large `before`/`after` JSONB snapshots ONLY in the single-entry `getAuditEntry` detail view (`includeData`). Status/reports/backup surfaces stay on `dev.console` / `backup.*`.
- **Alternatives considered:** (a) Gate the audit viewer on `audit.read` OR `dev.console` (the first Session-8 build, a literal reading of the NEXT_TASK "audit.read / dev.console" note) — the review flagged it: a `dev.console` grant given for operational *monitoring* would then double as a bulk export of every actor's IP / user-agent / email. (b) Drop the email from list rows too — makes the activity log useless (the "who" is the point). (c) Leave ip/ua on every list row — unnecessary PII surface for high-volume listing.
- **Why chosen:** The `audit_log` is the one table that aggregates PII (actor email + request `ip_address`/`user_agent`); a dedicated permission is the right gate, and a developer/super_admin holds it anyway (`grants_all` / ALL), so privileged users are unaffected. Showing ip/ua only in the deliberate single-row drill-down is standard data-minimization and keeps the bulk viewer from being an IP harvester. Aligning the route and the reader on one key removes the boundary-vs-service mismatch.
- **Trade-offs:** A `dev.console`-only operator can read system status but not the audit log — intended (least privilege). Slightly narrower than the NEXT_TASK's "audit.read / dev.console" wording; recorded here as the deliberate, more-secure reading.
- **Future impact:** The Session-9 audit-viewer UI requests `audit.read`; a future "who saw what" access-log, if needed, follows the same dedicated-permission + minimize-on-list pattern.

## DL-048 — The media-migration status is a PURE read (reusing `selectMigrationCandidates`), and the status endpoint DEGRADES GRACEFULLY on a cold/unreachable Neon instead of going dark.

- **Decision:** `status.mjs#getMediaMigrationStatus` computes the `/public`→Cloudinary plan/counts by reading the `media_asset` inventory and bucketing it with the EXISTING pure `lib/media/migrate.mjs#selectMigrationCandidates` — it does NOT invoke the gated `migratePublicAssets({dryRun:true})` mutator just to read (a read must not route through a `media.migrate` gate). For resilience: `checkDatabase` NEVER throws (a cold/suspended Neon is returned as `{ok:false, unreachable:true}` with a coarse, non-leaking message — the raw Prisma `P1001` host:port is logged server-side, not returned); `getSystemStatus` wraps each sub-check in `safe()`→`{error}`; `getInfraUsage` guards BOTH the `pg_database_size` query AND the `media_asset.aggregate` (degrading to nulls); `getDevConsoleReports` isolates the infra read to `{error}`; and `/api/dev/status` settles status + reports with `Promise.allSettled` so one failing half never discards the other. A migration-ledger read failure returns a distinct `{status:'ledger-unreadable', upToDate:null}` shape rather than mislabeling every applied migration as `pending`.
- **Alternatives considered:** (a) Call `migratePublicAssets({dryRun:true})` for the plan — couples a read to the `media.migrate` permission and the mutator's fail-fast config check. (b) `Promise.all` for status+reports (the first build) — the review proved that an unguarded `media_asset.aggregate` in `getInfraUsage` would throw on an unreachable Neon, reject the whole route, and return a blanket 500 in EXACTLY the cold-Neon scenario the monitor exists to surface. (c) Surface raw driver errors to the client — leaks the Neon host:port. (d) Diff migrations against an empty ledger on a read failure — reports a fully-applied schema as entirely un-applied.
- **Why chosen:** A monitoring surface must be MOST reliable when the thing it monitors is unhealthy; the resilience pattern (never-throw probe + `safe()` sub-checks + isolated reads + `allSettled`) keeps the DB-health payload reportable on a cold/suspended Neon (the documented Neon-latency reality). Reusing `selectMigrationCandidates` keeps one classification source of truth and avoids a needless permission coupling.
- **Trade-offs:** The status payload can carry partial `{error}` markers for sub-checks that failed — by design (a reported degraded state beats a thrown 500). `getMediaMigrationStatus`'s `alreadyMigrated` bucket counts non-archived migrated rows (matching the dry-run), reported alongside the pending counts.
- **Future impact:** The Session-9 dashboard renders the partial/degraded states directly; Session 10 can add alerting thresholds off `database.state` / `migrations.upToDate` / `media.fullyMigrated`.

## Session 9 — Admin Panel decisions

## DL-049 — Users & Roles is the ONE net-new backend; its mutations carry explicit privilege-escalation guards (developer-only flag AND grant; system-role protection; no self-lockout).

- **Decision:** Session 9's only new business-logic module is `lib/users/admin.mjs` (the UI calls existing Session 3–8 services for everything else). It manages `app_user` accounts, `role` definitions and `role_assignment` grants, following every established convention — authorize FIRST via `assertActorPermission` (system bypass / 401 / 403), one semantic `audit_log` row per op through the shared `auditedMutation` (using the operation-level `grant_role` / `revoke_role` actions), JSON-safe shaped returns (never raw rows / `passwordHash` / BigInt), and DB uniques honored via friendly mapped errors (`role_assignment_unique_active_grant_uq`, `role_key_key`, `app_user_email_key`). On top of that it adds four **privilege-escalation guards**: (1) only a DEVELOPER (`app_user.is_developer`) may create or set the `is_developer` flag — a `grants_all` super_admin holds every *permission* but is not a developer, so it cannot mint an account that bypasses RBAC; (2) **granting** a `grants_all` or `is_system` role (developer / super_admin) is likewise developer-only — holding `role.assign` alone must not let a super_admin delegate the unrestricted bypass to a puppet (this closes the parity gap the adversarial review found as a CRITICAL: the flag was guarded but the equivalent role-grant was not); (3) a new role can never be created with `grants_all` (reserved for the seeded developer), and SYSTEM roles are modification-protected except their `description` (so an admin cannot strip super_admin's permissions or deactivate it and lock everyone out); (4) you cannot suspend/disable your OWN account (self-lockout guard). Account-status changes need no session invalidation because authorization is resolved live per request (DL-019).
- **Alternatives considered:** (a) Gate user/role admin on a single coarse `admin` permission — loses least-privilege (a content editor's helper could escalate). (b) Guard only the `is_developer` flag (the first Session-9 build) — the review proved a non-developer with `role.assign` could still grant the `developer` role and obtain the identical `grants_all` bypass (`resolveEffectivePermissions` treats the flag and a grants_all role assignment identically), so the flag guard alone was a false sense of security. (c) Allow editing system roles' permissions (the seed's "super_admin is editable" note) — kept the data model editable but operationally protected the two system roles to remove the foot-gun.
- **Why chosen:** The `audit_action` enum already models `grant_role`/`revoke_role`, so RBAC changes get first-class, attributable audit rows for free. The escalation guards make the panel safe to hand to a `super_admin` who is NOT a developer: they can run the institute but cannot manufacture an unrestricted-bypass account. Live + static tests assert every guard (the DL-049 live test exercises the flag AND both grant paths).
- **Trade-offs:** Only a developer (or the seed, via `prismaBase`) can grant developer/super_admin; a super_admin manages custom non-system roles. Two genuinely distinct humans sharing an exact cleaned email can't both exist (citext unique) — acceptable. An internal `roleView` (ungated) is used by `createRole`/`updateRole` for their return + audit-before/after so they don't require the actor to ALSO hold `role.read` (and so the audit before/after are the same full shape).
- **Future impact:** Any future RBAC surface reuses this service + its guards; scoped (per-unit/per-year) grants are already supported by the `role_assignment` scope columns and the service (the panel grants institute-wide; the API accepts a scope).

## DL-050 — The whole admin write surface is ONE registry-driven, audit-wrapped endpoint (`POST /api/admin/action`); institute-wide ops gate a permission at the boundary, content/org ops use `requireUser()` + the service's own scope gate.

- **Decision:** Every admin mutation posts `{ action, args }` to a single route (`app/api/admin/action/route.js`) which `requireUser()`s (authentication + live active-account check) then delegates to `lib/admin/handlers.mjs#dispatchAdminAction`. The dispatcher is a **registry**: each action key maps to `{ permission | scoped | console, run }` and the matching EXISTING service call, executed inside `withAuditContext({ actorUserId, ipAddress, userAgent })` so the one semantic audit row each service writes is attributed. Gating is per-entry: **institute-wide** ops (year / media / users / roles) assert their single permission at GLOBAL scope at the dispatcher; **content / org / appointment** ops are `scoped: true` — the route only authenticates, and the underlying service authorizes at the item's true `(year, org-lineage)` scope (which the boundary cannot know a-priori for a unit-scoped grant), authorizing BEFORE any state read; **backup** ops are `console: true` and self-gate via `authorizeConsole` (any-of `backup.*` / `dev.console`). The route maps a thrown error to `{ error, code }` + status via the auth error shape or `mapDbError` (never leaking internals); the client IP is validated with `net.isIP` before it reaches the `inet` audit column.
- **Alternatives considered:** (a) One route file per resource/op (dozens of files) — repetitive and scatters the auth-first rule. (b) Server Actions — viable on Next 16, but the codebase's established pattern is API routes (`/api/events`, `/api/dev/*`) and a single route keeps the boundary identical and obvious. (c) Gate EVERY op (incl. content/org) on a global permission at the route — would lock out a legitimately unit-scoped editor, since a scoped grant doesn't satisfy a global-scope `can()` check; the service-scope gate is the correct authority for those.
- **Why chosen:** One choke point makes "authorize first, then audit, then delegate" a single, testable invariant (a static test asserts every registry entry declares a gate + a `run`). Reusing the services means NO new mutation/audit/visibility pipeline (the guard rail). The scoped-vs-institute-wide split is the precise reading of the "requireUser() + the service gate" guard-rail option.
- **Trade-offs:** A `scoped` entry relies on its service authorizing (verified: every CMS/org/appointment service authorizes before any state read); a regression there would be a hole, so the contract is documented at the registry. The dispatcher resolves permissions once per call (the page already resolved them for nav) — fine at admin-panel traffic.
- **Future impact:** A new admin operation is a registry row + (usually) an existing service; the Session-10 hardening can add rate-limiting / CSRF defense at the one route.

## DL-051 — The admin UI is gated Server Components + `router.refresh()` reads with a single client mutation path; the shared `lib/admin/*` helpers are PURE and client-safe (no prisma) so the client/server boundary never leaks.

- **Decision:** The panel renders as Next 16 Server Components gated by `lib/admin/server.mjs#loadAdminContext` / `loadModuleContext` (which never throw — they return `unauthenticated` / `inactive` / `no-access` / `ok` for the layout to branch on) over a permission-filtered nav (`lib/admin/nav.mjs#buildAdminNav`). Reads are server-side (`lib/admin/reads.mjs`, each asserting its module's read permission) and re-fetched after a mutation via `router.refresh()`; revision diffs are computed entirely client-side from server-loaded views via the PURE `view-models#diffViews` (no diff endpoint). The shared helpers — `nav.mjs`, `view-models.mjs`, `forms.mjs` — are deliberately **prisma-free and dependency-light** (nav restates the one-line `hasPerm`; forms restates the service regexes; view-models imports only the pure content-type catalog) so they are safe to import from Client Components AND unit-testable without a DB; `server.mjs` / `reads.mjs` are server-only (they import prisma, which would fail a client bundle — a natural enforcement). The form validators MIRROR the service validators (email / role-key / year-label / slug / password / media kinds) so the client shows inline errors before a round-trip without ever being the authority. (b) Putting view-model/nav logic inline in components — untestable without rendering React. (c) Importing `lib/rbac/authorize.mjs#can` into the nav helper — pulls prisma into a client-imported module; restating the 1-line additive check keeps it pure.

Pure shared helpers make the UI logic the thing the static suite actually tests (27 cases) and remove a whole class of "server-only code leaked into the client bundle" build failures.
- **Trade-offs:** A modal holds a snapshot until `router.refresh()` re-renders the page (the role-grant modal closes on success so it never shows stale roles); list editing for normalized payload children (mission points / meal timings) is deferred to the importer/API (a documented scope boundary). The double permission-resolve (layout nav + page module gate) is one extra cheap DB read per navigation.
- **Future impact:** New modules follow the same shape (gated Server Component → client actions → the one mutation route); Session 10 can add responsive/perf polish and prune the remaining static V1 pages now that `/admin` supersedes `app/admin/page.js`/`page2.js` (both removed this session).

## DL-054 — Deploy hardening: safe security headers (no CSP yet), SameSite=Lax + an explicit same-origin (CSRF) check, and a best-effort per-process rate limiter on the two write routes.

- **Decision:** (1) `next.config.mjs#headers()` sets `X-Content-Type-Options: nosniff`, `X-Frame-Options: SAMEORIGIN`, `Referrer-Policy: strict-origin-when-cross-origin`, `Permissions-Policy` (camera/mic/geo off), and HSTS on all routes. A strict **Content-Security-Policy is deliberately NOT set** — the public components inject inline `<style>` and the app relies on framework-inlined styles, so a strict CSP needs a nonce pipeline; it is the one documented remaining hardening item. (2) A new dependency-free `lib/http/guard.mjs`: `assertSameOrigin` rejects a cross-origin (or opaque/`null`) browser `Origin`/`Referer` on `POST /api/admin/action` and `POST /api/events` — defense-in-depth atop NextAuth's `SameSite=Lax` cookie; a request with NO Origin/Referer at all (a non-browser API client, which can't carry the Lax cookie) is allowed. `assertSameOrigin` allows the configured `NEXTAUTH_URL` host, so that env var must be the real public origin. (3) `makeRateLimiter` — a fixed-window in-memory limiter (admin 60/min/account, events 20/min) keyed by user-id-then-IP, returning a friendly 429 + `Retry-After`. The pure decision functions are unit-tested (`tests/security.test.mjs`).
- **Alternatives considered:** (a) A strict CSP now — high risk of breaking inline styles/`next/font`/`next/image` without a nonce pipeline; deferred. (b) No explicit CSRF check, rely on SameSite=Lax alone — Lax IS the real boundary, but an explicit same-origin check is cheap defense-in-depth and the conventional belt-and-suspenders for cookie-auth mutation routes. (c) Treat an opaque `Origin: null` as "absent/allow" (the first draft) — the review flagged that this silently no-ops the check for sandboxed-iframe/`data:` origins; changed to **reject** a present-but-unparseable Origin, only short-circuiting when both headers are genuinely absent. (d) A shared Redis/Upstash rate-limit store — over-engineered for this scale; the in-memory limiter is a coarse dampener.
- **Why chosen:** Maximum safe hardening without risking app breakage. Authorization is still the real lock (live per-request RBAC, DL-019); these headers/checks reduce the blast radius of clickjacking, MIME-sniffing, CSRF, and brute-force abuse.
- **Trade-offs:** The rate limiter is **per-process** — in a multi-instance/serverless deploy each instance has its own window, so it is a dampener, not a global quota (front it with a shared store for a hard limit; documented in the runbook + KNOWN_ISSUES). No CSP yet. A misconfigured `NEXTAUTH_URL` (≠ real origin) would 403 legitimate same-origin POSTs — documented in the runbook troubleshooting.
- **Future impact:** A CSP nonce pipeline and a shared rate-limit store are additive; new write routes wrap `assertSameOrigin` + `assertWithinRateLimit` the same way.

## DL-058 — The entire Session 11+ member platform is delivered behind ONE developer-controlled PLUGIN (`feature_flag`); gating reads fail CLOSED.

(c) A coarse permission instead of `is_developer` — a grants_all super_admin would then flip the platform, contradicting "developer turns it on" and the DL-049 parity.

## DL-059 — M0 auth pivot: email+password ONLY *within the plugin* (Google rejected when ON, kept when OFF — not hard-removed); forced first-login change via a JWT flag + a pure edge-middleware decision.

- **Decision:** Rather than statically deleting the NextAuth Google provider, M0 makes the provider **conditional on `GOOGLE_CLIENT_ID` being set** and **rejects Google sign-ins in the `signIn` callback when the `member_platform` plugin is enabled** (a live DB read, available on the Node auth runtime). So the member platform is email+password only, while a plugin-OFF deployment keeps the legacy Google path — consistent with DL-058's reversible cutover. Add `app_user.must_change_password` (+ `password_set_at`); admin-provisioned accounts are created with an initial password and `must_change_password=true`. `authorizeCredentials` returns `mustChangePassword`; the JWT carries it (set at sign-in, refreshed on `trigger:"update"` and the existing `isDeveloper` lazy-load); a NEW root `middleware.js` redirects any authenticated must-change user to `/account/password` for all but an allowlist. The routing decision is the PURE, unit-tested `shouldForcePasswordChange` (lib/auth/must-change.mjs) — middleware reads ONLY the JWT (no Prisma on the edge), so it is naturally inert when the plugin is OFF (no account is ever flagged must-change then). `changeOwnPassword` verifies the current password, enforces the policy, clears the flag, then the client calls `session.update()` so the token clears and the redirect stops.
- **Alternatives considered:** (a) Hard-remove `GoogleProvider` from `authOptions` — irreversible without a code change and breaks the plugin-OFF legacy path; the requirement ("email+password only") is scoped to the member platform, which the plugin owns. (b) Enforce must-change with a DB read in the middleware — Prisma does not run on the Edge runtime; gating on the JWT flag is edge-safe and self-disabling when the plugin is off. (c) Stuff the must-change state into a server-side session table — the project is JWT-only (DL-019); refreshing the token via `update()` is the idiomatic clear.
- **Why chosen:** Keeps the cutover reversible (DL-058) while satisfying "email+password only" inside the platform; the pure-helper-plus-thin-middleware shape matches the project's tested-pure-logic ethos and avoids an edge Prisma dependency; the JWT-flag approach reuses the established live-refresh pattern.
- **Trade-offs:** A valid pre-existing JWT for a must-change user is redirected on the *next* request (not instantly revoked), acceptable like DL-019. The Google rejection is enforced at `signIn` (not by dropping the provider), so the provider still appears configured when env is set.
- **Future impact:** M1 (status/access modes) and route-scoped RBAC extend the same middleware; a future SSO can be added as another provider gated the same way.

## DL-060 — The centralized `notification` request queue is introduced in M0 (the M7 table, M0 slice); reference ids come from a DB sequence; account existence is never leaked; only assign/resolve are audited.

- **Decision:** M0 needs a place for admin-mediated account-creation and password-reset **requests**, which the plan assigns to the M7 `notification` table. Rather than a throwaway table, M0 introduces the real centralized `notification` table now (forward-compatible; M7 extends it additively with labels/more types/keyset pagination). Each row carries a human, monotonic, DB-UNIQUE `reference_id` (e.g. `AR-00001` / `PR-00001`) minted from a raw-SQL `notification_ref_seq` sequence (race-free: the sequence is atomic and `reference_id` is also UNIQUE as a backstop). A `status` CHECK (`open|assigned|resolved|dismissed`) guards the workflow. The public create flows (`createAccountRequest` / `createPasswordResetRequest`) are unauthenticated (the route gates the plugin + CSRF + rate-limit) and DEDUP onto an existing open request per email so repeated submits don't flood. The forgot flow records whether the account exists for the fulfilling stakeholder but the **route always returns a generic success** — account existence is never leaked to the caller. Only the cross-stakeholder actions **assign** and **resolve** are audited (one semantic row each); the public create is itself the durable record (no auth actor), so `Notification`/`FeatureFlag` are added to `AUTO_AUDIT_SKIP` (no auto-audit double-write, semantic path still fires).
- **Alternatives considered:** (a) A throwaway `password_reset_request` table for M0, replaced in M7 — two pipelines, exactly what the guard rails forbid. (b) Reuse `verification_token` — opaque NextAuth tokens, no status/assignment workflow. (c) App-computed reference ids (counter) — racy; a DB sequence is the correct race-free monotonic source. (d) Audit the public create too — noise with a null actor; the row is the record.
- **Why chosen:** One table, no parallel pipeline (the spine rule); the sequence is the standard race-free id source; not leaking account existence is the safe default for a public forgot form; auditing only the stakeholder actions matches the "audit cross-stakeholder actions" rule without flooding the log.
- **Trade-offs:** The reference-id sequence is a raw-SQL object invisible to Prisma drift (catalogued like the other triggers/uniques). M7 will add fields (label, keyset) via a further forward migration. Dedup means a user spamming forgot gets one queue row (intended).
- **Future impact:** M7 generalizes this table (threshold alerts, more types, keyset pagination, a notifications dashboard); every M0 row already has a referenceable id and assignment tracking.

## DL-061 — Password policy is one client+server source (CSPRNG generator split server-only); reset is admin-mediated (no self-serve email link); `user.delete` added with DL-049 escalation parity.

- **Decision:** A single password POLICY (`lib/auth/password-policy.mjs`: ≥10 chars, lower+upper+digit, not all-one-char, ≤128) is the authoritative rule, validated **server-side** in `changeOwnPassword`/`setUserPassword`/`createUser`/bulk-CSV AND mirrored **client-side** (re-exported through `lib/admin/forms.mjs`). - **Alternatives considered:** (a) Keep only `MIN_PASSWORD_LENGTH=8` — too weak for the stated "policy validated client AND server side". (b) A self-serve email-link reset — explicitly out of scope (admin-mediated by design); avoids building an email-token pipeline and keeps a human in the loop. (c) Soft-delete (status only) instead of `deleteUser` — the operator asked to "delete users"; a real delete with guards + a before-snapshot audit row is provided (soft states remain via `setUserStatus`).
- **Why chosen:** One policy source prevents client/server drift (the DL-051 pattern); splitting the generator keeps the validator client-safe; admin-mediated reset matches the explicit requirement and needs no email-token machinery; `deleteUser` with DL-049 parity keeps the panel safe to hand to a non-developer super_admin.
- **Trade-offs:** Admin-set initial/temporary passwords and bulk-CSV passwords must also meet the policy (operator convenience trade for security; weak CSV rows are reported, not silently created). A hard delete nulls the deleted user's authorship on historical rows (the audit log retains the deletion event); a future "anonymize instead of delete" is additive.
- **Future impact:** M2 (categories + per-email overrides) layers on the same authorize-first user service; the generator + policy are reused by any future credential flow.

## Session 11 — Member Platform M2 (RBAC categories + per-email overrides + smart search) decisions

## DL-062 — Per-email permission OVERRIDES (`user_permission_override`, grant|deny, optional scope) REVISE DL-026 #8's no-deny rule: `resolveEffectivePermissions` applies the additive role union FIRST, then overrides, and DENY WINS — while the developer / `grants_all` bypass still short-circuits BEFORE overrides. A GRANT override carries a "you can only grant what you hold" escalation guard (DL-049 parity).

- **Decision:** Add a new `user_permission_override` table `(user, permission, mode grant|deny, org_unit_lineage_key?, academic_year_id?, reason?)` and extend the pure `resolveEffectivePermissions(user, assignments, scope, overrides)` so resolution is now: (1) `is_developer` → `grantsAll` short-circuit (overrides ignored); (2) compute the additive role union over in-scope active assignments; (3) if any in-scope assignment is a `grants_all` role → `grantsAll` short-circuit (overrides ignored); (4) ELSE apply in-scope overrides — add every `grant` permission, then delete every `deny` permission, so **DENY WINS** over a role grant AND over a grant override at any applicable scope, independent of array order. This **revises DL-026 #8** ("permissions are additive union … no negative/deny permissions"). The override scope reuses the SAME `inScope` predicate as `role_assignment` (NULL dimension = "all"; a narrower override doesn't apply to a broader request). The service (`lib/users/admin.mjs#setUserOverride/removeUserOverride`) authorizes the new `permission.override` permission FIRST, writes ONE semantic audit row (create/update/delete on `user_permission_override`, via the shared `auditedMutation`), upserts by `(user, permission, scope)` (the `user_permission_override_unique_uq` NULLS-NOT-DISTINCT unique is the target — one override per scope, grant/deny mutually exclusive there), and carries a NEW escalation guard: a **GRANT** override requires the actor to be a developer/`grants_all` OR to effectively hold that permission at the override's scope (`assertCanGrantPermission` → `userCan`), so a `permission.override` holder cannot grant themselves or a puppet a permission they don't have; a **DENY** (de-escalation) needs no such guard. The override FK to `permission` (not a free-text key) means an unknown permission can't be stored; the table is registered in `TABLE_BY_MODEL` for a stable audit `entity_type`.
- **Alternatives considered:** (a) Keep the no-deny rule and model restrictions only as narrower roles — cannot express "this one account, minus this one permission" without bespoke roles per exception (combinatorial). (b) Put grant/deny on `role_assignment` — overloads the role-grant table and can't deny a permission a *different* role grants. (c) Store the permission as a text key — loses FK integrity (a typo'd key silently never matches). (d) Let overrides restrict a `grants_all`/developer account — rejected: the unrestricted bypass must stay unrestricted (parity with DL-049/DL-058), and a stray deny on it would be a foot-gun (the developer is the recovery path). (e) Audit grant overrides only — denies are equally security-relevant cross-stakeholder actions, so both are audited.
- **Why chosen:** A single-permission grant/deny per email is exactly the operator ask ("add OR remove a specific permission from any single email") and the least-surprising extension of the additive model — deny-wins is the only sound precedence (a removal must beat an addition). Short-circuiting the bypass before overrides keeps the developer/`grants_all` semantics identical to every prior session. The grant escalation guard closes the obvious self-escalation hole a naive override service would open; deny needs no guard because removing a permission can never escalate. The NULLS-NOT-DISTINCT unique mirrors `role_assignment_unique_active_grant_uq`, so duplicate institute-wide overrides are a DB-level impossibility (proven by a live test).
- **Trade-offs:** One extra indexed `findMany` per non-developer authorization resolution (overrides by `user_id`); negligible and only for non-bypass users. A scoped override's precedence is purely subtractive at that scope — there is no "deny everywhere except here" sugar (compose a broad deny + a narrow grant; deny still wins on overlap, so that pattern needs distinct scopes). Overrides on a developer/`grants_all` subject are silently inert (documented).
- **Future impact:** M1 (status modes) and any scoped-route RBAC read the same resolver unchanged. A future "explain my permissions" view can surface the returned `denied` set. The override table is the seam for per-event / per-club fine-grained access in M3/M5 (scope columns already FK org-unit-lineage + year).

## DL-063 — RBAC "categories" are seeded ROLES (data), not a new entity: a six-rung stakeholder ladder (normal_user → admin) added to `ROLE_DEFS`, each non-system / non-`grants_all` with a default `role_permission` map; `admin` is computed as the full catalog minus the developer-only ops.

- **Decision:** Model the member-platform "categories" as ordinary `role` rows (the category IS the role; its key powers search/grouping), seeded via the existing `ROLE_DEFS` + seed loop — NO new table or column. Add six: `normal_user` (empty permission set — a member has no back-office permission; the category is for grouping + future member features), `co_coordinator` (draft a unit's content), `coordinator` (full content lifecycle + media = the editor set), `secretary` (coordinator + structure: org_unit.update + appointment.*), `staff` (central content/announcements + notification.read), and `admin` (the institute administrator). `admin`'s permission set is **computed** as `PERMISSION_KEYS` minus the developer-only ops (`dev.console`, `backup.create`, `backup.restore`, `media.migrate`) so it can never drift as the catalog grows. All six are `isSystem:false`, `grantsAll:false` (editable by `role.update` holders; never the bypass). The pre-existing `developer` (system, `grants_all`) and `super_admin` (system, `ALL`) are kept as the bootstrap roles; `content_editor`/`org_manager`/`viewer` remain as additional operational roles. A `category:true` marker on the new defs feeds `CATEGORY_ROLE_KEYS` (the search facet's source of truth); the seed ignores the marker (no schema column).
- **Alternatives considered:** (a) A separate `role_category` table / a `role.category` column — over-modeling; the requirement is literally "categories = roles (data)", and a role key already groups users. (b) Make `admin` an alias of `super_admin` — `super_admin` is the protected SYSTEM bootstrap role (un-editable); the assignable `admin` category must be operator-editable, so they are deliberately distinct. (c) Give `admin` the literal `ALL` (like `super_admin`) — would hand a non-developer admin the dev-console/backup/migrate ops; excluding those keeps least-privilege while still letting an admin run the institute (incl. users/roles/overrides, guarded by DL-049/062). (d) Give `normal_user` a token read permission — inaccurate; a member has no back-office permission, and reads are public anyway.
- **Why chosen:** Reusing roles-as-data means zero new pipeline (the spine rule), instant compatibility with `resolveEffectivePermissions`, scoped grants (coordinator→club, secretary→council via the `role_assignment` scope columns), and the search/grouping facet. Computing `admin` from the catalog removes a maintenance trap (a new permission auto-joins `admin` unless explicitly dev-only).
- **Trade-offs:** Two broad roles (`super_admin` system vs `admin` category) can look redundant; the system-vs-assignable distinction is the reason and is documented. A future permission that should NOT be in `admin` must be added to the small dev-only exclusion set.
- **Future impact:** Admin/dev create more categories through the existing role editor; M3 "custom roles" reuse the same `role` + `appointment.title_override` path. The ladder is the default grant target for the M0 account-provisioning flow.

## DL-064 — Email-format smart search: ONE pure predicate (`lib/users/search.mjs`, reusing `parseInstituteEmail`) is the authority for BOTH a debounced client filter AND the server `listUsers` refinement; the DB does only a coarse, index-friendly pre-filter (a leading email prefix + category join + status).

- **Decision:** Implement the M2 search as a pure, client-safe module `lib/users/search.mjs` (`matchesUserFilter` / `filterUsers` / `userFilterFacets` / `instituteEmailPrefix` / `normalizeLevel`) that reuses the M0 `parseInstituteEmail` (`<year><level u|p|r><branch><serial>@iitjammu.ac.in`). `matchesUserFilter` is the SINGLE source of truth for the predicate (year / level / branch / role-category / status / free-text `q`). The admin Users tab filters CLIENT-SIDE over the loaded list with a debounced search box + facet dropdowns (instant, no round-trip). The server `listUsers` accepts the same criteria and does a COARSE DB narrowing — status (direct), category (a `role_assignment` join on the role key), and an email `startsWith` prefix when `year[+level[+branch]]` form a *leading* institute-email prefix (e.g. `2023ume`) — then applies the SAME `matchesUserFilter` over the page so the server result is precise and identical to the client's (free-text `q` matches PER-FIELD email-OR-name, exactly the DB `where.OR`, so there is no client/server drift). Email stays the unique identifier (citext unique, unchanged).
- **Alternatives considered:** (a) Two filter implementations (a SQL `where` builder + a client filter) — drift-prone; the adversarial review found exactly such a drift in an early cut (a concatenated-field free-text match) and it was unified to per-field. (b) Pure server-side filtering over all rows — the email format's non-leading parts (branch without year) can't be a `startsWith`, and full-table scans don't scale; the coarse-prefix-then-precise-filter split keeps the DB query index-friendly. (c) A generated `parsed_*` column / a separate index — premature; the prefix `startsWith` on the existing email column covers the common (year-led) case, and the pure filter covers the rest. 
- **Why chosen:** One predicate, two call sites (client UI + server/API) is the DL-051 "pure, client-safe helper mirrored on both sides" pattern applied to search; reusing `parseInstituteEmail` keeps the email grammar in one place (M0). Client-side filtering over the already-loaded list is instant and the right default for the admin panel's scale; the server criteria + coarse prefix make the same filter usable by a future API/large-scale caller.
- **Trade-offs:** With a NON-prefixable filter (e.g. branch without a year), the server's coarse DB step can't narrow, so the precise filter runs over the first `take` rows — a documented bound (the admin UI sidesteps it by filtering client-side over the loaded set). Re-applying status/category in the precise filter is redundant with the DB clauses (idempotent, harmless) but makes `matchesUserFilter` the lone final authority.
- **Future impact:** M6 profiles / M8 analytics reuse the parser + facets; a future keyset-paginated user search keeps the same predicate and adds the cursor at the DB layer.

## Session 11 — Member Platform M1 (user status & access modes) decisions

## DL-065 — User status becomes three ACCESS MODES (active / inactive / revoked), forward-migrated via a CREATE-style enum swap + data backfill (never an init rewrite), and enforced LIVE at the login/session layer (not in the JWT).

- **Decision:** Reconcile the legacy `UserStatus` set `{active, suspended, invited, disabled}` down to the three M1 access modes `{active, inactive, revoked}` and define each mode's behavior precisely: **active** = full (log in + browse + own achievements + participate in events); **inactive** = log in + browse + own achievements but **cannot participate in events**; **revoked** = cannot log in at all, sees only the public site. The mapping is **suspended → inactive**, **disabled → revoked**, **invited → inactive** (an un-onboarded account with no usable credential can't authenticate anyway), recorded as a forward migration (`20260630160000_member_platform_m1`) that **creates a new enum type, retypes `app_user.status` through a `CASE` backfill, drops the old type and renames** — the init migration is never rewritten and `migrate reset` is never run (DL-027). Enforcement is LIVE in the auth/session layer (DL-019), NOT in the JWT: `lib/auth/options.mjs#authorizeCredentials` + the `signIn` callback reject `revoked` (via the pure `canLogin`) while admitting `inactive`; the back-office boundary `requireUser()` stays **active-only** (an inactive account has no back-office permissions anyway — the RBAC resolver already returns none for non-active — and a revoked account can't authenticate); the member boundary `requireMember()` admits active + inactive; and a new reusable **`assertCanParticipate`** capability (active-only) is the seam M5's event playground will gate registration/scoring/attendance on. The single source of truth for the matrix is the pure, client-safe `lib/auth/access.mjs` (`canLogin` / `canParticipate` / `canViewNormal` / `USER_STATUSES`), mirrored by the live server gates (the DL-051 pattern). The two duplicated `USER_STATUSES` constants (`lib/users/admin.mjs`, `lib/admin/forms.mjs`) now re-export from it.
- **Alternatives considered:** (a) `ALTER TYPE … ADD VALUE` for the new values + leave the old ones — Postgres cannot DROP an in-use enum value, so the retired vocabulary would linger forever and the UI/validators would have to special-case it; the type swap cleanly retires them. (b) A boolean pair (`can_login`, `can_participate`) instead of an enum — loses the single, legible status field the whole UI/search/audit already keys on, and three named modes are exactly the ask. (c) Make `requireUser()` admit `inactive` too — rejected: keeping the back-office boundary active-only is the smallest-blast-radius change (an inactive ex-coordinator is immediately out of the panel) and the member surface has its own gate. (d) Carry status in the JWT for an edge check — the project resolves authorization live per request (DL-019); a stale token must never grant access, so status is read live at each boundary (and a revoked user is also rejected at the next sign-in).
- **Why chosen:** Three named modes match the operator's mental model and the M5/M6 roadmap; the CREATE-style swap is the only non-destructive way to retire enum values; live enforcement reuses the established DL-019 live-RBAC posture (no session table, revocation effective next request) and the DL-051 pure-mirror pattern, so client and server never drift. Modelling `assertCanParticipate` now (before any event feature exists) means M5 reuses it unchanged.
- **Trade-offs:** The enum swap is a heavier migration than an additive one (it drops/recreates the type) — done once, guarded by a safe `CASE … ELSE inactive`. An `inactive` account that previously held a back-office role silently loses panel access (correct, but a behavior change from the old "suspended" which already blocked everything). A revoked user with a still-valid JWT is rejected on their next request, not instantly (the accepted DL-019 trade).
- **Future impact:** M5 gates participation via `assertCanParticipate`; M6 profiles read the same access description. Any future mode (e.g. `pending`) is an additive enum value + a row in the `access.mjs` matrix.

## DL-066 — Three surfaces (member / admin / developer) + scoped-route RBAC reuse the EXISTING resolver scope matching; the member boundary admits inactive while the back-office boundary stays active-only; pure surface/scope helpers mirror the live gates.

- **Decision:** Formalize the three surfaces as a pure routing decision (`lib/auth/access.mjs#resolveSurface`: `is_developer` → developer, else any admin-module visibility → admin, else member) consumed by a new non-throwing server loader `lib/member/server.mjs#loadMemberContext` (mirroring `lib/admin/server.mjs#loadAdminContext`'s discriminated states: `plugin-off` → 404, `unauthenticated`, `revoked`, `view-disabled`, `ok`) behind a minimal member view (`app/member`). **Scoped-route access** (coordinator → own club page; secretary → own council; staff → playground + central announcements) requires NO new mechanism: it is `requirePermission(key, { orgUnitLineageKey, academicYearId })` resolved by the existing `lib/rbac/authorize.mjs#inScope` against the `role_assignment` scope columns (a NULL grant dimension = "all"; a non-NULL must equal the request; a narrower grant does NOT satisfy a broader/unscoped request). M1 only adds the named seam `requireScopedPermission` (an explicit alias) + the pure `scopeMatches` (restating `inScope` for client-safe route guards/tests). The member normal-view boundary `requireMember()` admits active + inactive (rejecting revoked + the allow-normal-view-off case); the back-office boundary `requireUser()` stays active-only.
- **Alternatives considered:** (a) A `route_scope` table or a new per-route ACL — over-modeling; the `role_assignment` scope columns + the resolver already express per-unit/per-year grants exactly. (b) A single `requireUser()` for every surface — conflates the member experience (which must admit inactive) with the back office (active-only); two boundaries keep each correct. (c) Throwing guards for the member page — a signed-out/revoked visitor would see a 500; a non-throwing `loadMemberContext` (like `loadAdminContext`) renders a friendly state. (d) Build the rich member pages now — out of M1 scope (M3/M6); the member view is deliberately minimal (identity + access mode + the dashboard link the user is entitled to).
- **Why chosen:** Reusing the resolver's scope matching means the security-critical logic has ONE implementation (tested live in `m1.db.test.mjs` and purely in `access.test.mjs`), and the surfaces are a thin presentational layer over data the RBAC engine already returns. The non-throwing loader matches the panel's proven pattern.
- **Trade-offs:** `scopeMatches` restates `inScope` (a 6-line duplication) to stay client-safe/prisma-free — the same deliberate trade `lib/admin/nav.mjs#hasPerm` makes; a static test pins them to the same behavior. The member view is intentionally bare until M3/M6.
- **Future impact:** M3 club pages and M5 playground routes call `requireScopedPermission` with the unit's lineage; M6 profiles reuse `resolveSurface` + `describeAccess`. A future role (secretary/staff already seeded, DL-063) gets scoped route access for free.

## DL-067 — A per-account "allow normal (member) view" toggle (`app_user.allow_normal_view`), audited through the ordinary `updateUser` path, gating the member surface independently of status.

- **Decision:** Add `app_user.allow_normal_view boolean NOT NULL DEFAULT true`. When `false`, the authenticated **member (normal) view is withheld** even though the account can still log in — modelling the "allow normal view on first login" control the operator asked for (an account can be onboarded but not yet cleared for the member experience). It is read by the pure `canViewNormal(status, allowNormalView)` (login-able AND toggle-on; a revoked account is never admitted regardless) and enforced by `requireMember()` / `loadMemberContext` (→ a `view-disabled` state). It is set through the ordinary audited `updateUser` patch (one semantic audit row) and a first-class registry action `user.setAllowNormalView`, with the client mirror in `validateUserForm` + a checkbox in the admin Users modal. It carries **no escalation concern** (it only permits/withholds a member's own normal view — never a back-office capability), so unlike the developer flag it needs no developer-only guard.
- **Alternatives considered:** (a) Overload `status` with a fourth value for "can log in but no member view" — conflates two orthogonal axes (access mode vs. member-view eligibility); a boolean is the precise model. (b) A separate gated setter/route — unnecessary; `updateUser` already audits profile edits and gates on `user.update`. (c) Default `false` (opt-in) — would hide the member view from every existing account on cutover; defaulting `true` keeps current behavior and lets an admin withhold it deliberately.
- **Why chosen:** A single nullable-free boolean with a safe default is the least-surprising way to express "this account may not see the member view yet", it rides the existing audited mutation path (no new pipeline), and `canViewNormal` keeps the rule in the same pure matrix as the status modes.
- **Trade-offs:** One more boolean column on `app_user`; the toggle is currently the only consumer of `requireMember`'s `view-disabled` branch (M3/M6 member pages will reuse it). "On first login" is satisfied by checking the toggle on every member-view load (including the first), not by a one-shot flag.
- **Future impact:** M3/M6 member pages all sit behind `requireMember()`, so the toggle gates them uniformly; a future onboarding flow can flip it as the final provisioning step.

## Session 11 — Member Platform M7 + M8 spine (notifications/feedback + developer dashboard) decisions

## DL-068 — M8 developer dashboard: the Action Log / Change History is an EXPORT over the existing audit reader (PII-minimized, DL-047), and the destructive storage ops are a DEVELOPER-ONLY `storage.manage` permission (excluded from the `admin` category like backup.* / media.migrate).

- **Decision:** M8's "Action Log / Change History" tab adds NO new write/reader pipeline — it is a JSON/CSV **export** (`lib/devconsole/audit.mjs#exportAuditLog` + the pure `auditEntriesToCsv`/`csvCell`) layered on the Session-8 audit reader, gated on the SAME `audit.read` and carrying the SAME data-minimized columns as the list view (who/what/when + summary — NOT ip / user-agent / before-after, DL-047), capped at 5000 rows per export. A NEW `storage.manage` permission gates the per-table threshold/export/**truncate** ops; it is added to `ADMIN_ONLY_EXCLUDED` so the computed `admin` category never gets it — destructive storage management is developer/super_admin-only, exactly like `backup.*` and `media.migrate` (DL-063). `feedback.*` and `mail.*` are NOT excluded (admins run support + comms); usage analytics gate on `dev.console`.
- **Alternatives considered:** (a) A new "action log" table/writer — redundant; `audit_log` already records every mutation (DL-012/028). (b) Include before/after in the bulk export — re-introduces the bulk-PII-export risk DL-047 closed; the single-entry drill-down already shows it. (c) Gate truncate on `backup.restore` — overloads a backup permission with a different op; a dedicated `storage.manage` is clearer and still developer-only. (d) Let `admin` truncate tables — a non-developer admin must not be able to wipe a table; excluding it keeps the panel safe to hand to an admin (the DL-049 spirit).
- **Why chosen:** Export-over-the-reader keeps the "one audit pipeline" invariant; the column parity makes the dashboard's download exactly the safe list shape; a dedicated dev-only permission expresses "destructive storage management" precisely and rides the existing computed-admin exclusion set so it can't drift.
- **Trade-offs:** A very large audit history needs filter-narrowing to export older rows (the 5000 cap is documented in the UI). One more permission in the dev-only set.
- **Future impact:** A future scheduled export / retention job reuses `exportAuditLog`; new destructive ops gate on `storage.manage`.

## DL-069 — The centralized `notification` queue is GENERALIZED (M7) with a free-text `label`, keyset pagination, and a generic `createNotification` (deduped) for system producers — extending DL-060 with no parallel pipeline.

- **Decision:** Extend the M0 `notification` table/service (DL-060) rather than add anything new: a nullable `label` column (grouping facet, e.g. "accounts"/"security"/"storage"), a keyset-paginated reader `listNotificationsPage` (a stable **createdAt+id composite cursor**, since `notification.id` is a uuid, not monotonic), and a generic `createNotification({ type, label, title, body, entityType, entityId, data, dedupeKey })` for internal/system producers (the M8 storage monitor raises `threshold_alert`s through it). `dedupeKey` collapses repeat notices onto one OPEN row (matched on type + open status + `data.dedupeKey`) so a recurring monitor can't flood the queue. The account/reset request creators now set a `label`; the existing ref-id sequence + assign/resolve workflow + audit posture (only assign/resolve audited; the row IS the public-create record) are unchanged.
- **Alternatives considered:** (a) A new `alert`/`system_notification` table — a parallel pipeline the spine rule forbids; the queue already models labelled, ref-id'd, assignment-tracked rows. (b) An enum for `label` — premature; labels are an open, admin-meaningful grouping, so free-text + an index fits. (c) Offset pagination — drifts under inserts; keyset on (createdAt,id) is stable and matches the audit reader's keyset ethos. (d) A DB-level dedup unique for alerts — there is no natural (type,email) key for a threshold alert; app-level dedup on an open row + `dedupeKey` is the right grain.
- **Why chosen:** One queue, one workflow, one ref-id source — M7 is purely additive over M0. The composite cursor is the correct stable keyset for a uuid-PK table; `dedupeKey` is the minimal, race-tolerant flood guard for system producers.
- **Trade-offs:** The (createdAt,id) cursor assumes createdAt is set on insert (it is, `@default(now())`). Two notices with the same `dedupeKey` but both already CLOSED would create a new row (intended — a re-alert after resolution).
- **Future impact:** M3 club announcements / M5 event notices / threshold + usage alerts all post via `createNotification` with a label; a notifications dashboard reads `listNotificationsPage`.

## DL-070 — Feedback / support tickets are a STANDALONE table (the DL-038 rule), not CMS content — public create with a unique ref id (FB-NNNNN) + a CHECK-guarded status workflow + a pure client-safe validator (DL-051).

- **Decision:** Add a standalone `feedback` table (NOT a `content_type` — it is operational support data with its own lifecycle, the DL-038 "own table" rule) with a human DB-unique `reference_id` (`FB-NNNNN` from a raw-SQL `feedback_ref_seq`) so a user can reference a bug against any `component`/service id. The status workflow `open → triaged → in_progress → resolved | dismissed` is a DB CHECK; `category ∈ {bug,issue,query,suggestion}` is a CHECK. The PURE, client-safe `lib/feedback/forms.mjs#validateFeedbackForm` is the single validation source mirrored by the server service (DL-051). The public create (`createFeedback`) is unauthenticated (the route gates plugin + CSRF + rate-limit) and is itself the durable record (NOT audited); the cross-stakeholder `assignFeedback`/`setFeedbackStatus` are audited (one row each, gated `feedback.resolve`); reads are keyset-paginated, gated `feedback.read`. An authenticated submitter is linked by SESSION id (the route never trusts a body-supplied user id).
- **Alternatives considered:** (a) A `content_type='feedback'` — feedback has no draft/publish/version/visibility needs and shouldn't sit in the CMS visibility pipeline (DL-038 precedent: the Mongo `queries` collection was deliberately a standalone concern). (b) Reuse `notification` — notifications are the admin REQUEST queue; feedback is a distinct public support channel with its own category/status vocabulary (mixing them muddies both). (c) Trust a body `userId` for attribution — a forgeable claim; deriving it from the session is the only safe link.
- **Why chosen:** A standalone table is the precise model for support tickets (own ref-id series, own status machine), it follows the established DL-038 rule, and the pure-validator mirror keeps the public form honest without being the authority.
- **Trade-offs:** A second ref-id sequence (`feedback_ref_seq`) + a near-parallel assign/resolve shape to `notification` (deliberate — different domains). Anonymous submissions carry only an optional email.
- **Future impact:** M5 closure-report feedback + any component-tagged bug reporting reuse this table; a future SLA/triage view reads the keyset list.

## DL-071 — Hidden usage analytics: a minimal `page_visit` table (BIGSERIAL) written by a BEST-EFFORT, never-audited, never-throwing beacon; aggregates gate on `dev.console`.

- **Decision:** Add a `page_visit` table (BIGSERIAL PK — high-volume, monotonic, keyset-friendly like `audit_log`, DL-018) holding only `path` + a coarse `section` (first path segment) + an OPTIONAL `user_id` (SET NULL) + `created_at`. `recordPageVisit` is **best-effort**: it writes via `prismaBase` (bypassing the audit extension) and **swallows all errors** (a failed analytics write must never break a page render), and `page_visit` is in `AUTO_AUDIT_SKIP` so a visit is **NEVER audited** (it would bury the log + double every visit's write). The `/api/usage` beacon is same-origin + rate-limited (120/min/IP) + plugin-gated and always returns 204. Aggregates (`getUsageAnalytics`: top sections/paths over a window) gate on `dev.console` and degrade to zeros on a read error.
- **Alternatives considered:** (a) Audit page views through `audit_log` — catastrophic volume + noise; analytics is explicitly a separate, hidden concern. (b) Store full deep paths as the section — high cardinality; the first-segment "section" keeps the group-by meaningful. (c) A third-party analytics SDK — the requirement is a self-hosted, structured, downloadable signal the dev dashboard owns. (d) Block the render on the write — wrong: analytics must be fire-and-forget.
- **Why chosen:** Minimal + PII-light + crash-proof is the correct posture for hidden usage tracking; BIGSERIAL + indexes on (section, createdAt) / (createdAt) make the aggregates cheap; gating reads on `dev.console` keeps the signal developer-facing.
- **Trade-offs:** Section granularity is coarse (by design). `page_visit` is the realistic unbounded-growth table — which is exactly why it is the (only) truncatable one (DL-072).
- **Future impact:** M8 resource monitoring + future per-section dashboards read it; a retention job truncates it (DL-072).

## DL-072 — Per-table size monitoring + developer-set thresholds (NON-blocking) + export-then-truncate, with an ALLOWLISTED, confirm-gated truncate and a live-table-name SQL-injection guard.

- **Decision:** `lib/devconsole/storage.mjs` reads live sizes via raw `pg_total_relation_size` (`getTableSizes`, guarded → `[]` on a cold Neon), compares them to a per-table `table_threshold` (developer-set via `storage.manage`, audited), and `getStorageReport` FLAGS over-threshold tables **without blocking anything** (the site + feature keep working) and can raise a **deduped** `threshold_alert` notification (DL-069). `exportTable` dumps ANY table (JSON/CSV, capped) and records a `backup_record` ledger row; `truncateTable` is **destructive → developer-only `storage.manage`, requires `confirm:true`, and is restricted to an explicit `TRUNCATABLE_TABLES` allowlist (currently only `page_visit`)**. Both export and truncate build raw SQL only from a table name **validated against the LIVE list of public tables** (`assertRealTable`, plus an identifier regex) — closing SQL injection. Truncating `page_visit` cannot erase the audit trail of the truncation (the audit row lands in `audit_log`, a different table).
- **Alternatives considered:** (a) Block writes when a table is over threshold — the requirement is explicit: warn but keep working. (b) Allow truncating any table — a foot-gun that could wipe content/RBAC/audit; an allowlist of genuinely append-only log tables (page_visit) is the safe default, extendable only by a reviewed code change. (c) Parameterized identifiers — Postgres can't parameterize a table name, so validate-against-catalog + an identifier regex is the correct injection defense. (d) Truncate without an export — the ledger row + the export-first UX preserve a recovery path.
- **Why chosen:** Non-blocking thresholds match the stated behavior; the allowlist + confirm + dev-only gate + catalog-validation make a genuinely destructive op safe to expose; reusing `backup_record` keeps one ledger.
- **Trade-offs:** The allowlist must be widened by a code change (intended friction). Sizes are point-in-time. Export is capped (50k rows) — a huge table needs an out-of-band dump.
- **Future impact:** M8 resource monitoring extends `getTableSizes`/`getInfraUsage`; a retention policy truncates `page_visit` after export; adding a new truncatable log table is a one-line, reviewed allowlist change.

## DL-073 — Bulk mail: nodemailer is LAZY + INJECTABLE (no hard dependency at import), sends are rate-limited with progress accounting, and senders are restricted to an admin/dev-maintained allowlist; bodies/recipients are never stored in the audit log.

- **Decision:** `lib/mail/service.mjs#sendBulk` (gated `mail.send`) validates the `from` against an ACTIVE `authorized_sender` row (allowlist maintained via `mail.manage`, audited), normalizes/de-dupes recipients, and sends in rate-sized batches with an inter-batch pause — tracking progress through the PURE, client-safe `lib/mail/progress.mjs` (`chunk`/`remainingAllowance`/`mailProgress`/`normalizeRecipients`) so the "X of Y sent; you can send N more now" bar logic is unit-tested and shared with the UI. The nodemailer transport is **lazy + injectable**: the module imports fine WITHOUT nodemailer installed (it is only `import()`ed when a real send runs and no transport was injected), so the static suite + a plugin-off deploy never need it, and tests inject a fake transport + a no-op `sleep`. A bulk send writes ONE semantic audit row with only the accounting (`from`, subject, total/sent/failed) — **never the bodies or the recipient list** (PII/size). Initial account passwords still go via the institute's EXTERNAL mail, never here.
- **Alternatives considered:** (a) A hard `import nodemailer` at module top — would force the dependency into every build/test even when mail is unused and break the static suite (nodemailer isn't installed here). (b) No allowlist (any authenticated sender) — fails the requirement and risks spoofed institute senders; the allowlist is the control. (c) Log recipients/bodies for traceability — a PII + size liability; the accounting row + the transport's own logs suffice. (d) A real per-minute timer in tests — flaky/slow; an injectable `sleep` keeps the loop deterministic and fast.
- **Why chosen:** Lazy+injectable keeps mail an OPTIONAL, operator-configured capability (set `MAIL_*` + `npm install nodemailer`) without taxing the rest of the system; the pure progress helpers make the rate-limit/progress testable; the allowlist + minimal audit row match the security/compliance asks.
- **Trade-offs:** The per-process rate limit is a dampener, not a global quota (like the other limiters, KNOWN_ISSUES #34); a real send needs out-of-band VM config (documented). A failed individual send is recorded per-recipient but doesn't abort the run.
- **Future impact:** M0's external-mail step can later migrate here behind the allowlist; a future queue/worker reuses `progress.mjs` + the authorized-sender gate.

## DL-074 — `groupByWindow` is the one reusable past/current/upcoming primitive for windowed listings (announcements + events), kept PURE alongside `splitEventsByDate`.

- **Decision:** Add a pure, exported `groupByWindow(items, { now, fromKey, untilKey })` in `lib/events/public.mjs` that partitions any windowed items into `{ upcoming, current, past }` by a `[fromKey, untilKey]` date window vs `now` (started-or-undated-and-not-ended = current), each bucket sensibly sorted. It is the shared primitive the M7 "past/current/upcoming" announcement listing AND the events board (and M3 club announcements) reuse, sitting next to the existing `splitEventsByDate` so windowing logic is one tested rule, not re-derived per surface.
- **Alternatives considered:** (a) A bespoke window split per surface (events vs central vs club announcements) — duplication + drift, the exact trap `splitEventsByDate` already avoided. (b) Compute windows in SQL — the publish-window visibility is already DB-enforced; a pure presentational grouping over already-fetched rows is the right layer and is unit-testable. (c) Bake it into the announcement reader only — M3/M5 need the same primitive, so a generic helper is the spine piece.
- **Why chosen:** One pure, generic, tested windowing rule serves every current/future windowed listing with configurable field keys; it matches the project's "tested-pure-logic" ethos.
- **Trade-offs:** Callers map their own window fields onto `fromKey`/`untilKey` (a tiny adapter); "current" intentionally includes undated-start items (ongoing).
- **Future impact:** M3 club announcements + M5 event listings call `groupByWindow`; a future calendar view reuses it.

## DL-075 — Club/society/chapter MEMBERSHIP is a STANDALONE many-to-many between `app_user` and `org_unit_lineage` (the DL-038 "own table" rule), keyed to the LOGICAL unit (lineage), not a per-year `org_unit` — durable across years, one user in many clubs; a scoped `membership.manage` gates it, and an idempotent CSV importer syncs coordinator-submitted email lists (DL-031).

- **Decision:** Add a `club_membership` table `(user_id, org_unit_lineage_key, role?, status, joined_at, created_by)` with a UNIQUE `(user_id, org_unit_lineage_key)` (the importer's idempotency key) and a `status` CHECK (`active|inactive`). Membership FKs to `org_unit_lineage` (the durable logical-unit identity, DL-007), NOT to a per-year `org_unit`, so a membership survives the academic-year transition and one account can belong to many clubs/societies/chapters. It is OPERATIONAL relational data with its own lifecycle → a standalone table (the DL-038 rule), NOT CMS content. The service (`lib/memberships/service.mjs`) authorizes a NEW **`membership.manage`** permission SCOPED to the unit's lineage (`assertActorPermission(actor, "membership.manage", { orgUnitLineageKey, academicYearId: currentYear })`, i.e. `requireScopedPermission`/DL-066) BEFORE any account/membership disclosure; a coordinator scoped to club A's lineage manages A's roster and is 403 on club B. `addMembership` upserts by `(user, lineage)` (idempotent — re-adding updates role/status in place, the DB unique backstops it); `removeMembership`/`setMembershipStatus` gate at the membership's lineage. The bulk **CSV importer** `importClubMemberships({ orgUnitLineageKey|orgUnitId|slug, csv }, actor)` authorizes ONCE at the lineage, resolves each email → account (missing accounts are REPORTED, not created — a membership needs an existing M0 account), upserts the mapping via `prismaBase`, and writes ONE semantic summary `audit_log` row (created/updated/missing/failed counts). `ClubMembership` is in `AUTO_AUDIT_SKIP`; single add/remove/status changes each write ONE semantic row via `auditedMutation`. Member roster reads (`listMembershipsForUnit`, PII: emails) are gated `membership.manage`; a NON-PII active-member COUNT (`getMembershipCountForUnit`) is ungated for the public club page; a member's OWN clubs (`listUserMemberships`, called with their own id) are ungated (self). `membership.manage` is added to the `coordinator` + `secretary` category roles (and thus, computed, to `admin`).
- **Alternatives considered:** (a) A per-year membership FK to `org_unit` — would force re-adding every member each academic year (the Transition Wizard would have to copy memberships); the lineage FK makes membership durable, matching `role_assignment`'s lineage scoping. (b) Model membership as a `role_assignment` of a `member` role — overloads RBAC (a member has NO back-office permission; `normal_user` is empty by DL-063) and can't carry a free-text membership role/label or a soft `inactive` state; the roster is a distinct concern from authorization. (c) A `content_type` — membership has no draft/publish/version/visibility needs (DL-038). (d) Auto-CREATE accounts for unknown emails during import — a security foot-gun (account provisioning is the audited M0 flow); the importer reports missing emails instead. (e) Per-row audit on bulk import — noisy for a 100-member club; ONE summary row is the right grain (transition-wizard parity, DL-031).
- **Why chosen:** A standalone lineage-keyed M-M is the precise, durable model the operator asked for ("one user in many clubs … coordinators submit member email lists; admin bulk-uploads the mapping to sync"), reuses the existing lineage identity + scoped RBAC with zero new mechanism, and keeps membership cleanly separable from RBAC and CMS so the future AI layer can read a clean roster relation. Scoping `membership.manage` to the lineage is the DL-066 pattern applied to a new surface. Reporting (not creating) missing accounts keeps provisioning in the one audited M0 path.
- **Trade-offs:** A membership is year-agnostic, so "who was a member in year X" is not directly recorded (accepted — participation-per-year is M5 event data; membership is the standing relationship). The importer resolves emails sequentially (Neon latency) rather than in one query — acceptable at club scale. `getMembershipCountForUnit` is ungated (a bare count is not PII); the member ROSTER stays gated.
- **Future impact:** M6 profiles read `listUserMemberships` for "my clubs"; M5 can tag event organizers/participants by club membership; a future per-year membership snapshot is an additive column + a wider unique.

## DL-076 — Club sub-content (Miscellaneous MARKDOWN docs, club-specific announcements & events) are CMS `content_type` callers BOUND to the club via `content_item.orgUnitId` (DL-037 reuse); `club_doc` is a NEW content type that REUSES `page_block_payload` (no new payload table, DL-006), each doc minting its OWN lineage (DL-041).

- **Decision:** The club "Miscellaneous" markdown docs are a NEW `content_type='club_doc'` — year-scoped, ORG-BOUND — that REUSES the existing `page_block_payload` table (`blockKind='markdown'`, `body=markdown`). Adding it is pure DATA (a `CONTENT_TYPE_DEFS`/`content_type_def` row; the handler is auto-generated from the payload config) — NO new payload table and NO migration for the type (DL-006). Each doc mints its OWN content lineage (the importer/service omit `lineageKey`), so a club can hold MANY docs under `content_item`'s `UNIQUE(contentType, year, lineage)` (the DL-041 own-lineage-per-document pattern). Club-specific ANNOUNCEMENTS and EVENTS are the existing `announcement`/`event` content types BOUND to the club via the (already nullable) `content_item.orgUnitId` — no new type. All three are CRUD-ed through the Session-3 CMS service (`createDraft`/`editDraft`/`publish`/…) SCOPED to the club's lineage by the existing `content.*` permissions + the resolver's `inScope` match (DL-066) — a coordinator scoped to club A can write A's docs/announcements/events and is 403 on club B. The public club page reads them via `lib/org/docs.mjs#listClubDocs`, `lib/events/public.mjs#listClubEvents/listClubAnnouncements` (archive-like — `enforceWindow:false` — so past + upcoming both show, partitioned by `splitEventsByDate`/`groupByWindow`).
- **Alternatives considered:** (a) A standalone `club_doc` table — throws away draft/publish/version/audit/visibility the CMS spine already provides; docs ARE editable, versioned content (unlike memberships), so the DL-037 "content_type caller" rule applies, not DL-038. (b) A new `club_doc_payload` table — unnecessary; `page_block_payload` (blockKind/body/data/primaryMedia) already models a markdown block, and `team_page`/`page_block` already share it. (c) Reuse the unit's `lineageKey` for its docs — `content_item`'s per-year lineage unique would cap a unit at ONE doc (the exact bug DL-041 caught for resources). (d) A new "club announcement" content type — needless; binding the existing type via `orgUnitId` reuses the whole events/announcements read+shape layer and the publish window.
- **Why chosen:** Maximum reuse of the spine (a module is a `content_type` + payload table + handler, cap 9): club docs get the full CMS lifecycle for the price of one seed row, and club announcements/events reuse the entire Session-6 read layer. Own-lineage-per-doc is the only model that allows many docs per unit under the existing unique. Binding via `orgUnitId` also makes M5's richer event↔organizer linking additive.
- **Trade-offs:** A `club_doc`/club-announcement/club-event lineage is year-local (a fresh year gets new lineage; the Transition Wizard's by-lineage copy won't carry it across years) — accepted, identical to resources/events (DL-041). The generic Session-9 admin editor is the CRUD surface for these (org-bound content needs an org-unit selection in the payload input — an editor-UX refinement, not a backend gap).
- **Future impact:** M4 Wall-of-Fame achievements and M5 problem statements follow the same content_type-caller pattern; a per-club content index reuses `listClubDocs`/`listClubEvents`/`listClubAnnouncements`.

## DL-077 — Markdown is rendered by ONE pure, dependency-free, ESCAPE-FIRST renderer (`lib/markdown/render.mjs`) — HTML is escaped before any markup is generated (injection is structurally impossible) and link hrefs are scheme-validated; reused by M4.

- **Decision:** Render markdown (club docs now; M4 achievement blocks later) with a pure, client-safe `renderMarkdown(md)` that (1) HTML-escapes the ENTIRE input FIRST (`escapeHtml`: `& < > " '`), then (2) applies a minimal, well-known markdown subset (ATX headings, fenced code, unordered/ordered lists, blockquotes, hr, paragraphs; inline bold/italic/code/links) over the already-escaped text. Because escaping precedes markup generation, there is NO un-escaped user text anywhere in the output, so raw HTML / `<script>` / event-handler attributes can never survive — injection is structurally impossible rather than filtered. Link hrefs are additionally scheme-validated (`isSafeHref`: only `http(s)://`, `mailto:`, and same-origin `/…`/`#…`; control chars + whitespace stripped so `java\tscript:` can't bypass) — an unsafe URL renders the link TEXT with no `href`. The safe HTML string is fed to `dangerouslySetInnerHTML`. No external markdown/sanitizer dependency (keeps the bundle + build light; `markdownPreview` gives a plain-text card excerpt).
- **Alternatives considered:** (a) `react-markdown` + `rehype-sanitize` — a real dependency (bundle + build weight + supply-chain surface) for a small, controlled subset; the escape-first renderer is ~120 lines, unit-tested, and provably safe. (b) Escape AFTER applying markdown (the common footgun) — lets raw `<script>` in the source survive into the output; rejected. (c) A blocklist/DOMPurify pass on generated HTML — escape-first is a stronger allowlist posture (nothing un-escaped ever exists) and needs no DOM. (d) Render markdown to React nodes (no `dangerouslySetInnerHTML`) — more code for no additional safety once escape-first holds.
- **Why chosen:** Escape-first is the one markdown approach that is safe BY CONSTRUCTION, not by the completeness of a filter; keeping it pure + dependency-free matches the project's tested-pure-logic ethos and the "no unnecessary deps" posture, and makes it trivially reused by M4. Scheme-validating hrefs closes the only injection vector escaping alone leaves (a live `javascript:` link).
- **Trade-offs:** The supported markdown subset is intentionally small (no tables/images/nested lists) — sufficient for club docs / achievement copy; extend the pure renderer (with tests) if richer content is ever needed. `dangerouslySetInnerHTML` is used, but only on output that is provably escaped-then-limited.
- **Future impact:** M4 Wall-of-Fame markdown/markdown+image blocks reuse `renderMarkdown` + `markdownPreview`; any future markdown surface (M5 problem statements/closure reports) uses the same one renderer.

(b) A separate `announcement_sync` table — a whole table for one boolean is over-modeling. (c) Duplicate a synced club announcement as a second central content_item — content duplication + drift. (d) Filter in SQL — the flag is on the payload (loaded after the header), so filter it where audience is already filtered (the shaping layer). (e) Default club announcements TO central — the operator asked for OPT-IN sync, so club-only is the correct default.

- **Future impact:** M5 event notices / a future digest reuse the same opt-in-to-central pattern; a role-aware widened central view is additive.

## Session 11 — Member Platform M4 (Wall of Fame / student achievements) decisions

## DL-080 — Achievements are a NEW `content_type='achievement'` (year-scoped, NOT org-bound) via the CMS spine (DL-037), with their OWN `achievement_payload` table holding typed scalars + a `blocks` JSONB of HYBRID ORDERED blocks; blocks are validated/normalized by a pure client-safe module through a generic-handler `coercePayload` hook, and markdown is rendered by the escape-first `renderMarkdown` (DL-077 reuse).

- **Decision:** Model the Wall of Fame as a NEW `content_type='achievement'` — `isYearScoped:true`, `isOrgBound:false` — driven entirely through the Session-3 CMS service (createDraft/editDraft/publish/…), so it gets draft/publish/version/audit/visibility for free (the DL-037 "new module = a content_type caller" rule). It gets its OWN payload table `achievement_payload` (1:1 with `content_revision`), NOT a reuse of `page_block_payload`, because an achievement is a distinct first-class entity with typed, queryable scalars (`category`, `achievement_date`, `hero_media_id`) — the same posture as `event_payload`/`announcement_payload` (DL-037). The HYBRID ORDERED blocks (markdown / markdown+image / banner / link / gallery) are stored in a `blocks` **JSONB** column (the DL-016 JSONB-block precedent) — heterogeneous, nested, ordered content that does not fit flat 3NF child columns (a gallery holds many media ids; a link has url+label). A pure, client-safe `lib/achievements/forms.mjs` (`normalizeBlock`/`normalizeBlocks`/`normalizeAchievementPayload`) validates + normalizes blocks (drops unknown keys, caps counts, coerces the date, reuses `isSafeHref` for link urls) and is mirrored on the client + server (DL-051). It runs on the server via a NEW optional `coercePayload(payload)` hook on the generic content-type handler config, invoked inside `writePayload` on BOTH create and edit — so the achievement's block schema is enforced (throwing a 422 `CmsValidationError`) with NO parallel content pipeline. Markdown block bodies are stored RAW and rendered by the escape-first `renderMarkdown` (DL-077) at the presentation layer — no HTML is ever produced/stored upstream.
- **Alternatives considered:** (a) Reuse `page_block_payload` (like `club_doc`, DL-076) — a `page_block_payload` row is 1:1 with a revision and holds ONE block, so it can't hold an ordered LIST of heterogeneous blocks without stuffing everything into its `data` JSONB and losing the typed `category`/`date` scalars; a dedicated table is cleaner and queryable. (b) A normalized `achievement_block` child table (like `mess_meal_timing`) — heterogeneous block fields (gallery = N media, link = url+label, banner = media+caption) make a flat child row sparse/awkward; JSONB is the right fit for a small, ordered, varied block list (DL-016). (c) Validate blocks in a separate achievement service that wraps createDraft — a parallel pipeline; the `coercePayload` hook keeps achievements on the ONE CMS service. (d) A markdown/sanitizer dependency — DL-077's escape-first renderer already makes injection structurally impossible; no new dep.
- **Why chosen:** A dedicated payload table + a JSONB block list is the normalized-where-it-matters, flexible-where-it-must-be design the protocol favors: typed scalars for reads/filtering/M6, a JSONB for the open-ended block content, and the whole thing rides the CMS spine (one seed row + one payload table + the auto-generated handler + the `coercePayload` hook). Reusing `renderMarkdown` + `isSafeHref` means the achievement copy inherits M3's proven safety with zero new surface.
- **Trade-offs:** The `blocks` JSONB is not individually queryable in SQL (accepted — the Wall of Fame is browsed as whole entries, not queried by block); block media referenced by id are resolved in the read layer (one batched query). The `coercePayload` hook is a small addition to the shared handler (guarded — only `achievement_payload` sets it), so other types are unaffected.
- **Future impact:** M5 problem statements / closure reports reuse the same hybrid-block + `coercePayload` pattern; a richer admin block editor renders `getContentTypeFieldSpec` + the pure `normalizeBlocks`.

## DL-081 — Achievement ↔ contributor mapping is a STANDALONE `achievement_credit` table (the DL-038 "own table" rule) crediting one achievement to a MEMBER *or* a CLUB (`org_unit_lineage`), each row exactly ONE target (a DB CHECK), durable so a member's and a club's contributions are trackable across a year (feeds M6).

- **Decision:** Add an `achievement_credit` table `(id, achievement_item_id → content_item, user_id? → app_user, org_unit_lineage_key? → org_unit_lineage, role?, sort_order, created_by)`. Each credit references EXACTLY ONE of `{user_id, org_unit_lineage_key}` — enforced by a raw-SQL CHECK `((user_id IS NOT NULL)::int + (org_unit_lineage_key IS NOT NULL)::int) = 1` — so one row is unambiguously a MEMBER contribution or a CLUB/stakeholder contribution. Two per-target UNIQUE indexes (`(achievement_item_id, user_id)`, `(achievement_item_id, org_unit_lineage_key)`) prevent duplicate credits (Postgres NULL-distinct means the two never collide). Clubs are referenced by the DURABLE `org_unit_lineage` (DL-007), and members by `app_user` — both durable ids — so credits survive an academic-year rollover even though the achievement `content_item` is year-scoped; M6 reads a member's / a club's contributions across years by these keys. `lib/achievements/credits.mjs#setAchievementCredits(itemId, credits, actor)` REPLACES the full credit set idempotently (dedup by target in code + the DB uniques as backstop), inside ONE transaction, writing ONE semantic `audit_log` summary row. Emails resolve to an EXISTING account (missing → 422, never auto-created — the M0 rule, mirroring memberships). `AchievementCredit` is in `AUTO_AUDIT_SKIP` (audited semantically here). `on delete cascade` from `content_item` (a hard delete of an achievement removes its credits; content is normally soft-archived), `restrict` from `org_unit_lineage`.
- **Alternatives considered:** (a) Two separate tables (`achievement_member_credit` + `achievement_club_credit`) — doubles the read/write surface for what is one "contribution" concept; one table with a one-target CHECK is simpler and lets a single ordered `sort_order` interleave members + clubs. (b) A single nullable-both design without the CHECK — allows ambiguous/empty credits; the CHECK makes the invariant a DB guarantee (spine posture: integrity is the DB's job). (c) Bind the achievement to ONE club via `content_item.orgUnitId` (like `club_doc`) — can't credit MULTIPLE clubs or ANY members, which the requirement explicitly needs; hence not-org-bound + a mapping table. (d) Credit the achievement's document lineage instead of the year's `content_item.id` — the achievement is a specific year's Wall-of-Fame entry, so the year `content_item` is the correct anchor; durability comes from the member/club target ids, not the achievement row.
- **Why chosen:** A standalone, polymorphic-but-CHECK-constrained mapping is the precise model for "a member's AND a club's contributions, trackable across a year" — one clean relation the future AI/M6 layer reads by `user_id` or `org_unit_lineage_key`, reusing durable identities and the audited replace-set pattern (transition-wizard/membership parity).
- **Trade-offs:** A credit is anchored to the year's achievement `content_item`, so "all of a club's achievements across years" is a union over years' credits (fine — each year's Wall of Fame is a distinct set). The replace-all API rewrites the credit set on each save (one summary audit row) rather than per-credit diffing — the right grain for a curated list.
- **Future impact:** M6 member/stakeholder profiles aggregate `achievement_credit` by `user_id` / `org_unit_lineage_key`; the per-club Wall-of-Fame slice (DL-082) is a read over club credits.

## DL-083 — The generic CMS `writePayload` uses `UPDATE` (not `upsert`) on a partial EDIT (`isCreate:false`) — the payload row always pre-exists on every edit path, and Prisma statically requires the `upsert.create` branch to carry every NOT-NULL payload column (a latent M3 bug surfaced by the first live run of `tests/m3.db.test.mjs`).

- **Decision:** In the generic content-type handler's `writePayload`, branch on `isCreate`: on create/copy/restore (`isCreate:true`, the WHOLE payload is supplied) keep the create-or-replace `upsert`; on a partial edit (`isCreate:false`) use `tx[model].update({ where:{revisionId}, data: scalars })`. This is correct because the payload row ALWAYS pre-exists on every edit path — `createDraft` writes it (`isCreate:true`), and `editDraft` either edits an existing draft (row created at draft creation) or opens a new draft via `copyPayload` (`isCreate:true`) BEFORE applying the partial patch; `restore` uses `copyPayload`. The prior unconditional `upsert` broke a partial edit of any type whose Prisma model has a NOT-NULL scalar (e.g. `announcement_payload.body`): Prisma STATICALLY validates the `upsert.create` shape and rejects it for the missing required column, even though only the `update` branch would ever execute. This surfaced as a `PrismaClientValidationError` in `tests/m3.db.test.mjs` (the club-announcement sync test edits `{syncToCentral:true}` alone) on the FIRST live run of the M3 suite (which was written-but-unrun under KNOWN_ISSUES #42).
- **Alternatives considered:** (a) Keep `upsert` and make every payload column nullable in Prisma — weakens the schema + doesn't fix the general class (any future required column re-breaks it). (b) Merge existing values into the create branch before upsert — an extra read + still wrong if the row genuinely didn't exist with required fields absent. (c) Make `body` specifically nullable — a point patch that leaves the latent bug for the next required-field payload. `update`-on-edit fixes the whole class and matches the real invariant (the row exists on edit).
- **Why chosen:** It matches the actual data-flow invariant (a payload row exists by the time any edit runs), fixes the entire class of "partial edit of a type with a required payload column", and is a minimal, well-commented change to the ONE shared handler. A defensive P2025 (record-not-found) on the impossible "edit a revision with no payload row" is an acceptable, clear failure.
- **Trade-offs:** If some future caller expected `writePayload(isCreate:false)` to CREATE a brand-new payload row, it would now 404 — but no caller does (verified: createDraft/editDraft/restore all ensure the row first). The in-memory test fake gained a faithful `update` method.
- **Future impact:** All M4 achievement edits + every existing type's partial edit now go through `update`; M5's richer payloads inherit the corrected behavior. (Recorded under M4 because the M4 work re-ran the M3 suite that surfaced it.)

- **Why chosen:** Satisfies "all data types in cards" + "arranged responsively" with a pure server render, reusing the existing `shapeResource` fields (`fileUrl`/`fileKind`/`externalUrl`/`resourceKind`) and the brand styling.
- **Trade-offs:** Thumbnails are unoptimized `<img>` (small, lazy-loaded; the source is already Cloudinary `f_auto,q_auto` for migrated assets); the icon set is fixed to the known kinds with a `file` fallback.
- **Future impact:** The `resolveKind` + `KIND_META` map is the single place to extend when a new `ResourceKind`/`MediaKind` is added.

## DL-101 — WALL-OF-FAME credits get a UI: the previously API/CLI-only `achievement.credits.set` is now editable on the achievement content-item editor — a member(-by-email)/club(-by-lineage) credit grid that bulk-replaces the credit set.

- **Decision:** An achievement is a `content_type='achievement'` content item edited on `/admin/content/[id]`; its relational CREDITS (`achievement_credit`, member OR club, DL-081) had a service (`setAchievementCredits`) + action but NO client. The content-item page now, for achievements, loads the current credits (`listCreditsForAchievement`, ungated) + the year's club/council lineage options (`loadAdminOrgUnits`, best-effort so a missing `org_unit.read` degrades to member-only crediting, never a page crash) and renders a new `AchievementCredits` section in `ContentItemClient`. The editor lists current credits, adds member (email) or club (lineage dropdown) credits with an optional role, and "Save credits" posts `achievement.credits.set` with the full desired set (the service is a bulk idempotent REPLACE authorized by `content.update` at the achievement's year scope).
- **Alternatives considered:** (a) a bespoke Wall-of-Fame admin module — REJECTED (an achievement is already CMS content; the credits belong on its editor); (b) inline crediting on the public `/wall-of-fame` page — deferred to the later inline-edit-on-public-page work; (c) an email free-text for clubs — REJECTED (a lineage dropdown is unambiguous; a manual key input is only the fallback when org units can't be loaded).
- **Why chosen:** Reuses the existing service/action/audit unchanged; the editor lives exactly where the achievement is already edited; best-effort loads keep it robust for content editors who lack org read.
- **Trade-offs:** The club dropdown lists the achievement-year's clubs/councils (crediting is by durable lineage, so the current-year display name is shown); crediting a member requires an existing account (the service rejects an unknown email, by design — accounts aren't auto-created).
- **Future impact:** The load-relational-data-for-a-content-type pattern on the generic content editor is reusable for any content type with a side table (e.g. event organizer tags could get the same treatment).

## DL-102 — BULK permission overrides: the per-email override editor is now a grant/deny CHECKBOX GRID that applies the whole institute-wide set in ONE call (`permission.override.setBulk` → `setUserOverrides`), replacing the one-permission-at-a-time dropdown.

- **Decision:** Replace the single-select `OverridesModal` (choose one permission + a grant/deny dropdown, apply, repeat) with a per-module checkbox grid: each catalog permission has mutually-exclusive Grant/Deny checkboxes (both off = inherit from roles). A GRANT checkbox is offered only for a permission the admin holds (`hasPerm`, mirroring the server escalation guard). Saving posts one `permission.override.setBulk` action → a new `lib/users/admin.mjs#setUserOverrides(userId, entries, actor)` that diffs the desired INSTITUTE-WIDE set against the account's current institute-wide overrides and reuses the existing `setUserOverride` (flip/no-op + escalation guard + one audit row each) and `removeUserOverride` for those no longer desired. SCOPED (unit/year) overrides are shown read-only and left UNTOUCHED (they remain API-managed, DL-062). Role-permission editing was ALREADY a checkbox grid (RoleModal), so this closes the remaining dropdown the request referred to.
- **Alternatives considered:** (a) loop `permission.override.set` from the client per change — REJECTED (N toasts + N `router.refresh`es, non-atomic UX); (b) one giant transaction in the bulk service — REJECTED per the Neon-latency rule (DL-031): the bulk applies each change via the proven per-item path sequentially (each its own audit row, semantically correct); (c) a tri-state `<select>` per permission — REJECTED (the request explicitly asked for checkboxes); (d) let the bulk also manage scoped overrides — REJECTED (scoping needs unit/year pickers; out of scope, and silently touching scoped grants would be surprising).
- **Why chosen:** One save for many permissions (the requested UX), atomic from the operator's view (one API call, one refresh), and it REUSES the escalation guard + audit of the per-item service so security/attribution are unchanged; client-side grant-gating prevents most escalation attempts before they reach the server (which still enforces).
- **Trade-offs:** The bulk issues one `role_assignment`-free `userPermissionOverride` write per CHANGED permission (not one row for the batch) — acceptable and correctly audited; if an admin attempts to grant a permission they don't hold (only reachable by editing the request), that one change 403s server-side and the rest still apply (partial), matching the per-item guard; clearing a restriction requires an explicit interaction.
- **Future impact:** `setUserOverrides` is the reusable bulk-override primitive; the grant-gated checkbox grid is the template if scoped-override editing is ever added (add unit/year pickers + pass the scope through).
